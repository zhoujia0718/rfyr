2问题详细分析

严重
P1
vip_tier 与 memberships 数据不一致

问题描述

系统存在两个存储会员状态的地方：users.vip_tier 和 memberships 表。这两个数据源之间没有强一致性保证，可能出现以下情况：

  • memberships 表有记录，但 users.vip_tier 未更新
  • users.vip_tier 显示 yearly，但 memberships 记录已过期
  • cancel 操作中，memberships 删除后 users.vip_tier 更新失败
  • 续期时，新旧记录处理不当导致多条 active 记录

代码证据

activate API 中使用 UPDATE + INSERT 混合方式，且 cancel 操作中删除 memberships 后更新 users.vip_tier 存在时序风险。

✅ 解决方案

引入数据库触发器或 Supabase RPC 函数，保证 memberships 表状态变更时自动同步 users.vip_tier。使用事务确保原子性。考虑使用 INSTEAD OF 触发器或外键约束。

创建 RPC: activate_membership(user_id, plan_type, days) 和 cancel_membership(user_id)，所有业务逻辑在服务端 RPC 内原子执行。

严重
P2
会员类型命名混用（5种命名方式）

问题描述

代码中存在 5 种会员类型的命名方式：

  • 'monthly' / 'yearly' — lib/membership.ts (客户端)
  • 'monthly_vip' / 'annual_vip' — Admin upgrade 操作
  • 'free' / 'monthly' / 'yearly' / 'permanent' — api-types.ts
  • 'none' — users.vip_tier 空值
  • 'monthly' / 'yearly' — memberships.membership_type

影响

  • hasPermission 检查时 'monthly_vip' 不在 PERMISSIONS 数组中，导致月卡用户无法访问
  • isYearlyMember 检查 membership_type === 'yearly'，但 upgrade 后存储为 'annual_vip'
  • 前端和后端类型不一致，TypeScript 类型形同虚设

✅ 解决方案

统一会员类型枚举，在 lib/constants.ts 中定义单一真实来源：

MEMBER_TIERS = ['free', 'monthly', 'yearly', 'permanent'] as const

所有 upgrade/downgrade 操作统一转换为标准命名。memberships.membership_type 也统一使用此枚举。建立类型守卫函数 isValidMembershipType(val) 进行运行时校验。

高
P3
支付系统未真正集成微信支付

问题描述

PaymentDialog 使用 setTimeout(3000) 模拟支付成功，5分钟倒计时和二维码展示都是 UI 模拟，并未真正调用微信支付 API。

  • 没有统一下单接口调用
  • 没有微信支付回调 (callback) 处理
  • 没有订单状态与支付结果的同步机制
  • 模拟支付按钮直接调用 activate，跳过了真实支付验证

现状

整个支付流程实际上是"假支付→直接激活"的模式，无法处理真实付费用户。

✅ 解决方案

接入微信支付 JSAPI 或 H5 支付：

1. 后端调用微信统一下单 API，生成 prepay_id
2. 前端使用 WeChat JSAPI 调起支付
3. 配置微信支付回调地址，接收异步通知
4. 回调验证签名后，调用 membership activate API
5. 移除模拟支付逻辑，区分测试环境与生产环境

高
P4
Paywall 配额计算逻辑分散在多处

问题描述

配额检查逻辑分散在 3 个地方，且计算方式不完全一致：

  • Paywall.tsx — effectiveFreeLimit, effectiveMonthlyLimit 计算
  • use-reading-limit.ts — maxCount, effectiveDailyLimit 计算
  • WechatGuideOverlay.tsx — maxCount, remaining 计算（另一套逻辑）

风险

当修改默认配额时，需要同时修改 3 处代码，容易遗漏。不同组件可能因缓存或时序问题返回不一致的配额判断结果。

✅ 解决方案

在 lib/reading-limit.ts 中建立单一配额计算中心：

getEffectiveQuota(user, settings) → QuotaResult

所有组件调用此统一函数获取配额信息。QuotaResult 包含：canRead, remainingReads, remainingDaily, isOverLimit, effectiveLimit, reason。ReadingSettings 表作为单一配置来源。

高
P5
reading_settings 表设计为单行不合理

问题描述

reading_settings 表主键固定为 'global'，整张表只有一行数据。这种设计混淆了"配置表"和"单值常量"的概念。

  • 无法支持多配置方案（A/B 测试、不同用户群体差异化配置）
  • 更新配置需要 UPDATE 行，而非 INSERT，缺少版本历史
  • 表级 RLS 策略无意义（所有用户共享同一配置）

更好的设计

将配额配置内嵌到 user_profiles 表的 JSONB 列中，或使用 key-value 表支持动态扩展。

✅ 解决方案

方案A：将 guest_read_limit, monthly_daily_limit 作为 user_profiles 的可选列，允许个性化配置。
方案B：使用 JSONB 列 user_profiles.quota_settings，存储 { guest_limit: 3, monthly_daily: 8 }，保留全局默认值 fallback。

中
P6
游客追踪依赖 IP+UA 哈希不稳定

问题描述

游客阅读追踪使用 IP + User-Agent 哈希作为 guest_id，存储在 guest_reads 表。但这个标识符极不稳定：

  • 用户切换 WiFi/移动网络 → IP 变化 → guest_id 变化
  • 浏览器升级 → UA 变化 → guest_id 变化
  • VPN/代理用户 → IP 频繁变化
  • 同局域网多用户 → 同一 IP → guest_id 冲突

后果

游客配额限制完全失效。用户只需切换网络或重启路由器即可绕过限制。

✅ 解决方案

方案A（推荐）：强制要求登录后才可阅读任何内容，彻底消除游客追踪需求。
方案B：使用 First-Party Cookie (localStorage) + 浏览器指纹作为辅助标识，不依赖 IP。
方案C：引入 Supabase Anonymous Sign-in，为每个游客分配临时匿名用户 ID，在用户登录时可合并历史记录。

中
P7
每日配额重置逻辑存在时区边界 Bug

问题描述

配额重置逻辑声称使用"中国标准时间 (CST, UTC+8)"，但代码实现中存在时区不一致：

  • 某些地方使用 new Date().toLocaleString('zh-CN', { timeZone: 'Asia/Shanghai' })
  • 某些地方使用 new Date().getHours() + 8 简单计算
  • Supabase 数据库存储为 TIMESTAMPTZ (UTC)，查询时需要转换

风险

在 UTC 23:00-24:00（即 CST 07:00-08:00）期间，配额可能异常重置两次或完全不重置。

✅ 解决方案

统一使用 date-fns-tz 或 moment-timezone 处理时区：
const CSTDate = zonedTimeToUtc(localDate, 'Asia/Shanghai')
将所有日期比较统一在 UTC 时间轴上完成，仅在 UI 展示层转换为中国时间。

中
P8
Admin 操作缺乏操作日志和审计

问题描述

Admin 可以执行 renew/cancel/upgrade/downgrade 等敏感操作，但没有任何日志记录：

  • 谁在什么时间操作了哪个用户的会员状态
  • 操作前的值和操作后的值是什么
  • 无法追溯异常操作（如给大量账户开通会员）

✅ 解决方案

创建 membership_audit_log 表：
(id, admin_id, target_user_id, action, old_value, new_value, created_at)
在每个 admin 操作中自动插入审计日志。定期检查异常模式（如短时间内大量续期操作）。

中
P9
会员激活后前端刷新机制不可靠

问题描述

PaymentDialog 调用 activate API 成功后，通过以下方式刷新状态：

  • 直接调用 refreshMembership() → 调用 GET /api/membership/status
  • 监听 rfyr:auth-refresh 自定义事件

问题

这两个刷新机制都是"尽力而为"模式：
— 刷新失败不重试
— 刷新时机依赖 API 响应速度（可能存在缓存延迟）
— 多个组件可能持有不同步的会员状态副本

✅ 解决方案

activate API 返回完整的最新 membership 数据，前端直接使用返回值更新状态，而非再次请求：
const newMembership = await activate(); setMembership(newMembership);
同时引入 React Query / SWR 进行全局状态管理，确保所有订阅方自动同步。

低
P10
缺乏会员到期提醒机制

问题描述

会员到期前/到期时没有任何提醒，用户可能因忘记续费而失去访问权限。

✅ 解决方案

方案A（推荐）：Supabase Cron Job 定时任务，每天检查即将到期会员（end_date 在 3 天内），发送邮件提醒。
方案B：在用户登录时检查 membership 状态，剩余 ≤7 天时显示横幅提醒。

低
P11
缺少会员权益试用体验

问题描述

没有"免费试用"机制。免费用户必须先付款才能体验付费内容，这会降低转化率。

✅ 解决方案

实现"新用户三日体验"：注册后自动获得 3 天年度会员权限，触发升级引导。试用结束前 24 小时发送邮件提醒。试用期间不限制阅读次数，仅在试用结束后展示付费墙。
