  • 前端：Next.js 16 + React 19 + TypeScript
  • UI：Radix UI + Tailwind CSS 4
  • 后端：Next.js API Routes (Route Handlers)
  • 数据库：Supabase (PostgreSQL)
  • 邮件：Resend API
  • 支付：Mock（无真实集成）

用户角色

  • 游客（未登录）— 只能看免费文章
  • 普通用户 — 免费3篇（含邀请奖励）

登录认证流程（密码登录）

用户切换到「登录」Tab → 填邮箱+密码

点击「登录」→ 调用 supabase.auth.signInWithPassword({email, password})

获取真实的 Supabase session（access_token, refresh_token, expires_at）

调用 persistLoginSession() → localStorage + cookie

触发 rfyr:auth-refresh → AuthProvider 刷新 → MembershipProvider 刷新

Session 持久化逻辑

localStorage['custom_auth'] = JSON { user: {id, email, ...}, session: {access_token, refresh_token, expires_at}, loginTime }

cookie: admin-session-local = encodeURIComponent(JSON { userId, email, loginTime })

loginTime 为 Unix 秒级时间戳（用于判断 session 有效性）

两种 Token 机制并存：Supabase 真实 token + HMAC 签名 fake token

退出登录流程

点击 Header 下拉菜单「退出登录」→ handleLogout()

清除 localStorage: custom_auth, membership, isLoggedIn, userEmail

清除 cookie: admin-session-local (max-age=0)

调用 supabase.auth.signOut()

setIsLoggedIn(false) → window.location.reload()

安全漏洞与逻辑问题分析

安全
⚠️ 以下安全问题按严重程度分级。Critical（严重）需要立即修复，High（高危）建议尽快处理，Medium（中危）和 Low（低危）可在后续迭代中优化。

🔴 Critical — 严重漏洞（必须立即修复）

V-C-01: 明文密码存储在 pending_registrations 表 严重

问题描述：用户注册时填写的密码以明文形式存入 pending_registrations 表，直到验证码验证通过后才创建用户。期间如果数据库被攻破，所有待验证密码均已泄露。

影响范围：所有在 pending_registrations 表中有记录的用户（注册未完成的）。

根因位置：app/actions/auth.ts 第426行附近，sendEmailVerificationCode 函数中 upsert pending_registrations 包含明文 password 字段。

修复建议：改为哈希存储：password_hash = createHash('sha256').update(password + salt).digest('hex')。验证时哈希后比对。注意：这样需要修改 pending_registrations 表结构。

V-C-02: 兑换码删除无权限验证 — 任何人可删除任意兑换码 严重

问题描述：DELETE /api/admin/redeem?id={id} 虽然有 requireAdmin(request) 校验，但该 API 端点接收 URL 查询参数 id，若攻击者能获取到兑换码的数据库 ID（通过猜测或信息泄露），即可构造请求删除任意兑换码。更严重的是，删除操作无任何审计日志。

影响范围：所有兑换码可能在不知情的情况下被删除。

根因位置：app/api/admin/redeem/route.ts DELETE 方法。

修复建议：添加兑换码 ID 的范围校验（确认兑换码属于当前管理员创建的）、添加操作审计日志、考虑软删除（标记为 deleted 而非物理删除）。

V-C-03: 开发者登录API暴露在生产环境风险 严重

问题描述：/api/dev/login 仅通过 NODE_ENV === "production" 判断是否返回错误。在某些部署场景下（Next.js 的 standalone 模式、docker 容器），NODE_ENV 可能被错误配置为 development，导致开发者登录接口对公网开放。

影响范围：如果部署配置错误，生产环境用户可无限制访问测试用户账号登录。

根因位置：app/api/dev/login/route.ts 第13行。

修复建议：将此 API 路由文件在生产构建时排除，或改为环境变量控制（NEXT_PUBLIC_DEV_LOGIN_ENABLED），确保生产环境默认关闭。

🟡 High — 高危问题（建议尽快处理）

V-H-01: 阅读记录写入存在竞态条件（TOCTOU） 高危

问题描述：/api/articles/[id] 和 lib/reading-limit.ts 中的 recordVisit 使用 SELECT→UPDATE 两步操作而非原子操作。两个并发请求可能同时通过超限检查，导致实际阅读次数超过上限。

触发条件：用户快速连续点击两篇文章，或使用脚本同时发送多个请求。

根因位置：app/api/articles/[id]/route.ts 第212-249行；lib/reading-limit.ts 第101-142行。

修复建议：使用 Supabase RPC 函数做原子 upsert，或使用 PostgreSQL 事务 + SELECT FOR UPDATE 锁。

V-H-02: X-User-Id Header 降级信任风险 高危

问题描述：getUserIdFromBearer() 在 Supabase token 验证失败后，会降级接受仅带 X-User-Id header 的请求（仅检查 UUID 格式）。如果攻击者能获取到任意用户的 UUID，即可冒充该用户身份调用 API。

影响范围：所有使用 getUserIdFromBearer 的 API 端点，包括阅读记录、会员状态、邀请统计等。

根因位置：lib/server-auth-user.ts 第139-147行。

修复建议：移除 X-User-Id 降级信任，或改为：仅当该 header 来自可信服务端代理时（检查特定 IP/IPv6）才接受。

V-H-03: 验证码 6 位数字碰撞概率高 高危

问题描述：注册验证码使用 Math.random() 生成 6 位数字（100000-999999），共 900000 种可能。若攻击者对同一邮箱在验证码有效期内（10分钟）暴力枚举，最坏情况需尝试 90 万次。但限流机制仅限制 60 秒内的发送次数（发送后可无限验证尝试）。

影响范围：通过邀请链接的特殊场景（referrer_code + referrer_article_id 组合）可能帮助攻击者定位目标邮箱。

根因位置：app/actions/auth.ts 第394行，Math.floor(100000 + Math.random() * 900000)。

修复建议：使用 crypto.randomUUID() 或更安全的随机数生成器；增加验证失败锁定机制（5次失败后锁定15分钟）。

V-H-04: 邀请奖励可被刷 — self-referral 检测薄弱 高危

问题描述：createReferral 仅检查 referrer.user_id === refereeId（不能邀请自己）。但攻击者可使用多个邮箱互相邀请，每个新账号都给邀请人增加 bonus_read_count。配合一次性邮箱，可低成本刷取大量阅读奖励。

影响范围：所有非会员的免费阅读上限可被绕过。

根因位置：lib/referral.ts 第47-50行。

修复建议：添加邀请关系上限（每人最多邀请 N 个有效用户）；添加 IP + 设备指纹检测；被邀请人必须完成特定操作（如阅读满3篇）才给邀请人奖励。

🟠 Medium — 中危问题

V-M-01: 阅读限制前端绕过 — Paywall 仅视觉模糊，内容仍在 DOM 中 中危

问题描述：Paywall.tsx 使用 blur-sm pointer-events-none 视觉遮挡文章内容，但 HTML 源代码中文章内容完整存在。攻击者可通过查看源代码或禁用 CSS 绕过付费墙。

实际风险：服务端 API 层已做强制限制（/api/articles/[id] 返回 401/403），前端视觉遮挡仅为 UX 增强，不会导致实际内容泄露。

根因位置：components/paywall.tsx 第108-110行。

说明：由于服务端已强制限制，此问题实际风险较低，仅作为代码质量建议。

V-M-02: /api/reading-settings PUT 缺少参数白名单校验 中危

问题描述：/api/reading-settings PUT 方法只检查了三个具体字段，未使用参数白名单。若数据库表结构扩展了新字段，恶意管理员可通过此 API 写入任意字段数据。

根因位置：app/api/reading-settings/route.ts 第60-68行。

V-M-03: Supabase Anon Key 暴露在前端代码 中危

问题描述：NEXT_PUBLIC_SUPABASE_ANON_KEY 被打包进前端 JS bundle，对所有用户可见。虽然 RLS（行级安全策略）应保护数据，但如果 RLS 配置不当，攻击者可利用 Anon Key 直接操作数据库。

根因位置：lib/supabase.ts 第4-5行。

V-M-04: 会员状态前端缓存可能导致权限判断滞后 中危

问题描述：MembershipProvider 在初始化时使用 localStorage 缓存的 membership 数据（getMembershipFromStorage()），有效期长达 5 年。当会员到期后，前端可能在缓存过期前一直显示错误的会员状态。

⚪ Low — 低危/代码质量问题

V-L-01: 邀请码生成算法可预测 低危

问题描述：fake token 的 HMAC 签名使用 .slice(0, 16) 截断，降低了碰撞抗性。

V-H-06: getUserIdFromBearer 降级信任可被利用 高危

当 Supabase token 验证失败后，代码会降级接受仅带有 X-User-Id header（UUID 格式）的请求。这意味着：

① 如果攻击者获取到某个用户的 UUID（可通过各种 API 响应泄露），可以冒充该用户

逻辑漏洞与设计缺陷 未覆盖

会员有效期判断：memberships vs users.vip_tier 不同步

兑换码兑换时同时更新 memberships 表（第217-237行）和 users.vip_tier（第240-243行）。

但 isYearlyMember() 优先查 memberships 表（第31-46行），Fallback 查 users.vip_tier（第49-62行）。

如果两个表的 vip_tier 值不一致（例如 memberships 中是 monthly，users 中是 yearly），isYearlyMember 可能返回 true（memberships 优先），但实际 vip_tier 是 yearly。

反之亦然：兑换年卡后 memberships 写入了 yearly，但 users.vip_tier 可能因某些错误未更新。

月卡每日重置：跨服务器时区问题

所有重置判断使用 toLocalDateString()（北京时间 UTC+8）。

但 Supabase 存储的是 UTC 时间（数据库默认时区）。

如果服务端部署在非中国时区（如 UTC 或美国时区），toLocalDateString() 计算的"今天"可能与数据库存储的 last_read_date 日期不一致，导致跨日判断错误。

例如：北京时间凌晨1点，服务端服务器是 UTC-8，new Date() 返回昨天。

邀请奖励无限叠加风险（self-referral 加强版）

createReferral() 只检查：

① referrer.user_id !== refereeId（不能邀请自己）

② referrals 表中不存在相同 referrer_id + referee_id 组合

攻击方式：使用 3 个邮箱 A、B、C 创建循环邀请关系：A 邀请 B → B 邀请 C → C 邀请 A。每人都获得 bonus_read_count += 2。三人可以轮流互相邀请无限刷取。

检测循环邀请需要图遍历，复杂度 O(n)。

阅读设置缓存：多进程部署下 cache 不一致

lib/reading-settings.ts 第27-28行使用模块级变量 settingsCache 做服务端缓存。

在 Next.js 的 serverless 函数或 Docker 多实例部署时，每个进程/实例有独立的 Node.js 运行时，settingsCache 不共享。

管理员修改 reading_settings 后，一个实例更新了缓存，但其他实例的缓存仍是旧值，直到各自缓存过期（1分钟）。

虽然 clearSettingsCache() 在 PUT 时被调用（第86行），但只清理了当前进程的缓存。

兑换码有效期使用 ISO 时间字符串比较，精度不一致

redeemCode() 第154-158行：

const expiresAt = new Date(redeemCodeData.expires_at).getTime()

const now = Date.now()

expires_at 存储在数据库中可能是 TIMESTAMPTZ（带时区），PostgreSQL 存储时会转换为 UTC。读取时 JavaScript 的 Date() 会根据本地时区解释这个 UTC 时间。如果服务端时区不是 UTC+8，比较结果可能差几个小时。

架构设计问题 设计缺陷

认证系统三元并存：复杂度极高

当前系统存在三种认证机制：

① Supabase Auth（真实 JWT token，signInWithPassword）

② 自定义 fake token（HMAC 签名，email 注册后使用）

③ 纯数字账号登录（admin_token_ 前缀，假 session）

这三种机制在 persistLoginSession 中混用，session.access_token 可能是 pwd_ / admin_token_ / fake_ / cookie_ 等多种格式。getUserIdFromBearer 需要处理所有这些情况。

建议：统一为一种机制（Supabase Auth JWT）。

数据冗余：memberships + users.vip_tier + user_profiles.vip_status

同一个"用户是否是会员"的信息存储在三处：

① memberships 表（精确的起止日期）

② users.vip_tier（字符串标签）

③ user_profiles.vip_status（布尔值）

isYearlyMember 需要查 memberships 再查 users.vip_tier，MembershipProvider 需要调用 API 查 users.vip_tier，RedeemDialog 需要查 user_profiles.vip_status。

建议：统一使用 memberships 表作为唯一数据源，添加数据库函数获取会员状态。

多 Hook 数据源导致状态不一致

useArticleReader、useReadingLimit、useDailyQuotaCheck 都试图获取阅读数据，但各自独立调用 API。

服务端 recordUserRead 可能在 useArticleReader 请求完成后更新数据库，但 useDailyQuotaCheck 的缓存仍是旧值。

建议：使用 React Query / SWR 等数据获取库，统一缓存管理。

Mock 支付与真实支付流程混淆

PaymentDialog、redeemCode、memberships 表同时存在，意味着系统可能支持两种会员开通方式：兑换码和 Mock 支付。

但 Mock 支付仅写入 localStorage，不写数据库。如果用户清除浏览器数据，会员状态消失。

redeemCode 是真实流程，写入数据库。PaymentDialog 应仅作为 UI 示例，不应调用任何实际激活函数。
