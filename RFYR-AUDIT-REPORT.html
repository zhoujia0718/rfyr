P0-1
兑换码使用不安全随机数，可被暴力枚举
🔴 严重
文件
lib/redeem.ts 第 58–66 行
模块
兑换码系统
问题描述
兑换码使用 Math.random() 生成。Math.random() 是伪随机数生成器， 其输出在 JavaScript 中可被预测。攻击者可以在本地运行脚本，生成所有可能的兑换码组合 （RFYR-MONTH- 或 RFYR-YEAR- 前缀 + 6 位字符，共 32⁶ ≈ 10 亿种组合）， 然后批量向服务器发起兑换请求。
// 当前代码：Math.random() 生成 6 位字符后缀 58function generateRedeemCode(type: "monthly" | "yearly"): string { 59 const prefix = type === "monthly" ? "RFYR-MONTH" : "RFYR-YEAR" 60 const chars = "ABCDEFGHJKLMNPQRSTUVWXYZ23456789" 61 for (let i = 0; i < 6; i++) { 62 suffix += chars.charAt(Math.floor(Math.random() * chars.length)) 63 } 64 return `${prefix}-${suffix}` 65}
对业务的影响
直接经济损失。攻击者生成并兑换大量年卡兑换码，免费获得年度VIP会员， 解锁所有个股挖掘等高价值内容。或者批量兑换月卡后低价转卖，冲击正常销售渠道。

补充说明：虽然兑换码有 3 天有效期，但结合 IP 限流漏洞（见 P1-4）， 攻击者可用多 IP / 多账号分布式绕过，每秒尝试数千个码，3 天内可遍历相当一部分空间。
修复方案
改用 Node.js 内置的 crypto.randomBytes()（密码学安全随机数）：
1import { randomBytes } from "crypto" 2 3function generateRedeemCode(type: "monthly" | "yearly"): string { 4 const prefix = type === "monthly" ? "RFYR-MONTH" : "RFYR-YEAR" 5 const chars = "ABCDEFGHJKLMNPQRSTUVWXYZ23456789" 6 const bytes = randomBytes(6) // 密码学安全的随机字节 7 const suffix = Array.from(bytes) 8 .map(b => chars[b % chars.length]).join("") 9 return `${prefix}-${suffix}` 10}

app/admin/membership/create/page.tsx 第 28–33 行

管理员填写"手动开通会员"表单后点击提交，页面只弹出 alert('会员开通成功') 就跳转了。 完全没有发起任何 API 请求到服务端。 这意味着管理员操作的意图与实际效果完全不符——页面在欺骗管理员。

P1-2
邀请奖励更新非原子操作，并发邀请时奖励丢失
🟠 高
文件
lib/referral.ts 第 126–160 行
模块
邀请系统
问题描述
邀请奖励的更新分三步：① 读取当前奖励数量 → ② 计算新值 → ③ 写入。 如果用户 B 同时被 C 和 D 邀请，第②步读取的值都不包含对方的修改， 后写入的会覆盖先写入的，导致 C 的邀请奖励被 D 的写入覆盖、丢失。

对业务的影响
奖励莫名丢失。用户正常邀请了朋友，却没收到应有的阅读奖励， 引发用户投诉"邀请了怎么没奖励"。影响邀请活动的推广效果和用户信任。

P1-8
兑换操作非原子，服务崩溃时导致会员开通但兑换码未标记使用
🟠 高
文件
lib/redeem.ts 第 187–240 行
模块
兑换码系统
问题描述
一次兑换涉及 5 个独立的数据库操作（更新/插入 membership、更新 users、更新 user_profiles、更新 redeem_codes）， 不是事务。如果服务器在操作过程中崩溃（如数据库连接超时、内存不足）：
1
✅ 更新 memberships 表（会员开通成功）
2
❌ 更新 users.vip_tier 时数据库连接断开 → 操作中断
3
❌ 兑换码状态未更新 → status 仍为 unused
4
⚠️ 用户有了会员权限，但兑换码还能被再次使用
对业务的影响
同一兑换码被使用多次。用户免费获得多个月卡/年卡，重复享受会员权益。 在 Supabase Serverless 环境中，网络抖动、超时比较常见，此问题触发概率不低。
