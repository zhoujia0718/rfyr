格式
fake_{userId}|{expiresAt}|{HMAC64}
签名算法
HMAC-SHA256（完整 64 字符，不截断）
有效期
用户登录：7 天
管理员登录：7 天
真实 JWT 有效期
Supabase 默认：1 小时（无刷新）
flowchart TD subgraph GEN["生成 fakeToken"] G1[generateFakeToken userId] --> G2["expiresAt = now + 7天"] G2 --> G3["signature = HMAC-SHA256(userId|expiresAt, HMAC_SECRET)"] G3 --> G4["token = fake_{userId}|{expiresAt}|{sig64}"] end subgraph VER["验证 getUserIdFromBearer"] V1[读取 Authorization: Bearer token] --> V2{token 以 fake_ 开头?} V2 -->|否 真实JWT| V3[supabase.auth.getUser token] V3 -->|成功| VOK([返回 userId]) V3 -->|失败| VERR([返回 null]) V2 -->|是 fakeToken| V4[读取 X-User-Id header] V4 --> V5[verifyFakeTokenSignature\n格式 + HMAC + 过期 + userId匹配] V5 -->|通过| VOK V5 -->|失败| VERR end
已修复H2（多处）：原来读取 token 写死 authData.fakeToken，而实际 token 存在 session.access_token。现统一为 session?.access_token || fakeToken。
已修复多处：login-form、site-header、admin/login/page 原来用 pwd_${Date.now()}/cookie_${Date.now()} 作为占位 token，HMAC 验证必然失败。已统一改为空字符串 ''。
新发现fakeToken 无吊销机制（无黑名单）。用户主动退出后，已签发的 token 在 7 天内仍然有效，持有旧 token 的请求仍可通过认证。
3B — Admin Cookie 认证（admin-session-local）
flowchart LR A[Admin API 请求] --> B[requireAdmin 读 Cookie] B --> C[verifyAdminCookieSignature] C --> C1{Base64 可解码且4段?} C1 -->|是 新格式| C2["Base64解码: salt_userId_expiresAt_sig\n验 HMAC + 过期时间"] C1 -->|否 旧格式| C3["纯文本: userId_expiresAt_sig\n验 HMAC + 过期时间"] C2 -->|通过| OK([返回 null 认证成功]) C3 -->|通过| OK C2 -->|失败| ERR([401]) C3 -->|失败| ERR
已修复C1：原 requireAdmin 除验证 Cookie 外，还调用 Supabase DB 查管理员邮箱白名单，但因缺少 await 导致 data 始终为 null，所有 Admin API 均返回 401。现移除 DB 查询，仅验证 HMAC Cookie。
新发现旧格式 Cookie 解析用 split("_") 处理，但 UUID 中含 -（不含 _），因此 parts2[0] 实际上能正确取到 userId。但若未来 userId 格式变化（含 _）则会截断。新 Base64 格式无此问题。

新发现问题总览（20 项）
P1邮箱 Magic Link 登录 JWT 1小时过期，无 refresh_token 刷新逻辑，用户1小时后需重新登录。
P2文章请求每次创建 3 个独立 Supabase 客户端（共 5 次 DB 查询），应合并为单次查询。
P3/api/membership/status 只读 users.vip_tier，不检查 memberships.end_date，过期会员仍被视为有效。
P4hasFreeArticleAccess 功能无效：user_profiles.referrer_article_id 字段从未被写入。
P5月卡 recordUserReadAtomic 无乐观锁，并发请求可同时绕过 dailyLimit 检查。
P6GET /api/guest-reading 响应暴露 guestId（SHA256 哈希），生产环境应移除。
P7POST /api/guest-reading 无速率限制，变换 UA 可绕过游客配额无限阅读。
P8atomic_increment_counter RPC 若未部署，邀请奖励静默失败，无监控告警。
P9兑换码原子认领成功后若 upsert memberships 失败，码已标记 used 但用户无会员，无回滚。
P10fakeToken 无吊销机制，退出后 token 在 7 天内仍然有效。
P11useDailyQuotaCheck 独立维护 state，不使用 ReadingContext，可能显示不一致的配额数字。
P12会员状态本地缓存 TTL 24h，管理员降级后用户最多 24h 内仍享有旧权限。
P13fetchData 对 readCount（终身总计）直接 setReadCount，无 Math.max 保护，可被慢响应覆盖实时数据。
P141B 路径（registerUser）注册后未自动生成邀请码，与 1A 路径行为不一致。
P15pending_registrations 全量过期行无定时清理，长期积累垃圾数据。
P16guest_reads 过期行（expires_at）无自动清理机制，长期积累。
P17邀请码 8 位碰撞风险：约 4 万用户后碰撞概率 0.1%，INSERT 失败时无重试。
P18移除模拟支付后无真实回调接口，实际支付流程依赖管理员手动激活。
P19findAuthUserByEmail 最多遍历 6000 条 Auth 用户，大用户量时注册耗时严重。
P20Paywall 前端判断与服务端 API 强制之间存在时间差，可能出现显示与实际不一致。
