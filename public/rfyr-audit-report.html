二、安全问题（严重 → 高 → 中 → 低）

2.1 严重问题

🔴
S-1: 实盘记录 API 完全无认证，可被任何人 CRUD
/api/portfolio/route.ts 严重 数据安全
▶

问题描述：

/api/portfolio 的 GET/POST/PUT/DELETE 四个方法均未做任何身份验证。任何知道该 API 路径的人都可以：

  • 查询所有用户的实盘记录
  • 以任意用户身份创建虚假的实盘数据
  • 修改他人的实盘记录
  • 删除任意实盘记录
问题代码
// GET: 获取所有记录 — 任何人可查 export async function GET(request: NextRequest) { // ... 无任何认证代码 const { data, error } = await supabase .from('portfolio_records').select('*') ← 公开读取 // POST: 创建记录 — 任何人可写 export async function POST(request: NextRequest) { // ... 无任何认证代码 const { data, error } = await supabase .from('portfolio_records').insert([body]) ← 公开写入 }
修复建议
在所有方法开头添加认证检查：

export async function POST(request: NextRequest) { const userId = await getUserIdFromBearer(request) if (!userId) { return NextResponse.json({ error: '请先登录' }, { status: 401 }) } // 将 userId 作为 user_id 字段写入，确保只能创建自己的记录 body.user_id = userId // GET/PUT/DELETE 也需验证只能操作自己的记录 }
🟢
S-2: pending_registrations 表明文存储用户密码
app/actions/auth.ts ✅ 已修复 数据安全 / 合规
▶

修复状态：✅ 已修复（2026-04-18）

现在使用 SHA-256 哈希存储（salt: rfyr_salt_v1），验证时比较哈希值，不存储明文密码。

修复代码
// 存储时哈希 import { hashVerificationCodeSync } from '@/lib/security' // lib/security.ts — 哈希函数 export function hashVerificationCodeSync(code: string): string { const { createHash } = require("crypto") return createHash("sha256") .update(code + "rfyr_salt_v1") .digest("hex").slice(0, 16) }
新增文件
lib/security.ts — 包含 SHA-256 哈希和 HMAC 签名工具函数，有同步/异步双版本。
🔴
S-3: /api/debug/email 调试接口无任何认证，可任意创建用户
/api/debug/email/route.ts 严重 未授权操作
▶

问题描述：

POST /api/debug/email 可以无需任何认证即可调用，内部调用 supabaseAdmin.auth.admin.createUser() 创建任意邮箱的 Auth 用户。这是纯调试接口，应在生产环境完全禁用或添加 IP 白名单/密钥认证。

问题代码
// debugSendConfirmationEmail — 无任何认证保护 export async function debugSendConfirmationEmail(email: string) { const { data, error: createError } = await supabaseAdmin .auth.admin.createUser({ email, password: tempPassword }) ← 任意创建用户 ... }
修复建议
// 方案1：生产环境禁用整个路由 export async function POST(request: NextRequest) { if (process.env.NODE_ENV === 'production') { return NextResponse.json({ error: 'Not available' }, { status: 404 }) } // ... 原有逻辑 } // 方案2：添加密钥认证 const DEBUG_SECRET = process.env.DEBUG_SECRET if (request.headers.get('x-debug-secret') !== DEBUG_SECRET) { return NextResponse.json({ error: 'Forbidden' }, { status: 403 }) }

2.2 高风险问题

🟠
H-1: 管理员认证 Cookie 设置为 httpOnly: false，易被 XSS 窃取
/api/admin/login/route.ts 高风险 Cookie 安全
▶

问题描述：

管理员登录后设置的 admin-session 和 admin-session-local 两个 Cookie 均设置了 httpOnly: false，意味着 JavaScript 可以读取这些 Cookie。如果站点存在 XSS 漏洞，攻击者可以直接窃取管理员 Cookie 并冒充管理员。

问题代码
// 两个 Cookie 均 httpOnly: false ← 安全风险 response.cookies.set("admin-session-local", sessionPayload, { path: "/", maxAge: 60 * 60 * 24 * 7, sameSite: "lax", httpOnly: false, // ← 应设为 true }) response.cookies.set("admin-session", userId, { path: "/", maxAge: 60 * 60 * 24 * 7, sameSite: "lax", httpOnly: false, // ← 应设为 true })
修复建议
将 Cookie 改为 httpOnly: true，同时设置 secure: true（生产环境 HTTPS 下）。
注意：由于 admin 页面是纯前端 SPA，设置为 httpOnly:true 后，admin 页面通过 JavaScript 无法读取 Cookie，需改为服务端渲染或改用 localStorage（需同时防护 XSS）。

response.cookies.set("admin-session-local", sessionPayload, { path: "/", maxAge: 60 * 60 * 24 * 7, sameSite: "strict", // 改为 strict 防止 CSRF httpOnly: true, // 防止 JavaScript 读取 secure: true, // HTTPS only })
🟠
H-2: 邀请码校验 API 无认证，可被无限遍历暴力破解
/api/referral/validate/route.ts 高风险 暴力破解
▶

问题描述：

POST /api/referral/validate 完全无认证，可被任何人调用。通过遍历邀请码（8位字母数字组合），可以枚举出所有有效邀请码。虽有一定复杂度（约 2.8 万亿种组合），但结合社工或已知用户信息可大幅降低搜索空间。

问题代码
// 无认证检查，任何人可调用 export async function POST(request: NextRequest) { const { data, error } = await supabase .from("referrer_codes") .select("user_id") .eq("code", code.toLowerCase()) }
修复建议
添加速率限制或 IP 限制：
// 方案：添加 IP 级别速率限制（使用内存 Map 简单实现） const ipRequestMap = new Map<string, { count: number; resetAt: number }>() // 或使用 upstash/ratelimit 等专业库
🟠
H-3: 管理员 Storage 上传/读取接口无认证
/api/admin/storage-upload/route.ts, /api/admin/storage-file/route.ts 高风险 未授权文件操作
▶

问题描述：

Storage 文件上传（POST /api/admin/storage-upload）和文件读取（GET /api/admin/storage-file）接口完全没有认证检查。任何人都可以向这两个接口发送请求，上传任意文件到 Supabase Storage，或读取 Storage 中的任意文件。

现有防护

代码中限制了只允许 article-pdfs 和 article-images 两个 bucket，且路径检查了 .. 和绝对路径，防止路径遍历。但攻击者仍可上传恶意文件到允许的 bucket。

修复建议
在两个接口开头添加管理员认证检查：
// 读取 admin-session-local cookie 验证管理员身份 const adminCookie = request.cookies.get("admin-session-local") if (!adminCookie?.value) { return NextResponse.json({ error: 'Unauthorized' }, { status: 401 }) } try { JSON.parse(adminCookie.value) // 验证格式 } catch { return NextResponse.json({ error: 'Invalid session' }, { status: 401 }) }
🟠
H-4: getUserIdFromBearer 允许伪造 token + X-User-Id 绕过认证
lib/server-auth-user.ts 高风险 认证绕过
▶

问题描述：

getUserIdFromBearer 函数的认证逻辑过于宽松：

  • 以 pwd_/magic_ 开头的 token 会跳过 Supabase 验证，直接信任 X-User-Id header
  • 真实 token 验证失败后也会降级信任 X-User-Id header
  • 还支持 ?uid= query 参数作为兜底

只要攻击者知道（或猜测）一个用户 ID，就可以构造请求冒充该用户。

问题代码
// 伪造 token → 信任 x-user-id header（任意用户可冒充） if (/^(pwd_|magic_|magic_refresh_|pwd_refresh_)/i.test(token)) { const userIdHeader = request.headers.get("x-user-id") if (userIdHeader?.trim()) { return userIdHeader.trim() // ← 任意设置 userId } } // 真实 token 失败后也降级 → 仍可伪造 const userIdHeader = request.headers.get("x-user-id") if (userIdHeader?.trim()) { return userIdHeader.trim() // ← 降级信任 }
修复建议

这个机制是专为解决 Supabase session 不稳定而设计的"妥协方案"。建议改为：

  1. 只在使用真实 Supabase token 时信任 X-User-Id header（作为 cache busting）
  2. 伪造 token 场景下，改为从 custom_auth 中读取 userId（服务端无法读取 localStorage，所以需要换方案）
  3. 考虑改用真实 Supabase token 配合服务端 refresh 机制

2.3 中风险问题

🟡
M-1: Admin 页面仅客户端认证，可被直接绕过
app/admin/page.tsx 中风险 认证绕过
▶

问题描述：

Admin 页面的认证检查完全在客户端（React useEffect）进行，攻击者可以直接在浏览器开发者工具中注入 localStorage.setItem('custom_auth', '...') 或直接调用 React 的 setLoading(false) 方法绕过认证，直接访问 admin 页面内容。

问题代码
// 客户端认证 — 可被 JS 注入绕过 if (!isAuthenticated) { toast.error('请先登录') window.location.href = '/' // 仅是 JS 跳转，仍可访问页面 return } // ← 攻击者可在控制台注入 custom_auth 后刷新页面即可访问
修复建议
所有 Admin API 都应该做服务端认证，Admin 页面本身也应通过服务端渲染 + middleware 保护，或至少在 API 层做服务端认证。
🟡
M-2: 兑换码 DELETE 接口无认证，可被任意删除
/api/admin/redeem DELETE 中风险 未授权操作
▶

问题描述：

DELETE /api/admin/redeem?id=xxx 仅检查 URL 中的 id 参数，但完全无管理员认证。任何人可以删除任意兑换码记录。

🟡
M-3: 管理员登录缺少防暴力破解机制
/api/admin/login/route.ts 中风险 暴力破解
▶

问题描述：

管理员登录接口没有验证码、IP 限流或账号锁定机制。攻击者可以通过暴力破解或撞库攻击尝试登录管理员账号。结合 httpOnly: false 的 Cookie，一旦登录成功可立即获取 Cookie。

🟡
M-4: /api/referral/code 打印完整调试日志包含敏感信息
/api/referral/code/route.ts 中风险 日志泄露
▶

问题描述：

该接口在生产环境日志中打印了完整的 Authorization header、x-user-id header、cookie 和 uid 参数，可能暴露敏感认证信息。

export async function GET(request: NextRequest) { console.log("[Referral API] headers:", { authorization: request.headers.get("authorization"), // ← 敏感 x_user_id: request.headers.get("x-user-id"), // ← 敏感 cookie: request.headers.get("cookie"), // ← 敏感 })
🟡
M-5: sameSite: "lax" Cookie 易受 CSRF 攻击
/api/admin/login/route.ts 中风险 CSRF
▶

问题描述：

管理员 Cookie 设置为 sameSite: "lax"，允许来自其他域的 GET 请求携带 Cookie。虽然 POST 需要同源，但如果站点存在 GET 型 CSRF（如 JSON GET 请求），攻击者可利用跨站图片等方式触发管理员操作。

2.4 低风险问题

🔵
L-1: loginUser Server Action 可在不知道原密码的情况下重置密码
app/actions/auth.ts 低风险 业务逻辑
▶

问题描述：

loginUser 函数直接调用 updateUserById 设置新密码，不需要原密码验证。这意味着任何人如果知道一个用户的 username/phone，就可以修改该用户的密码（虽然该函数签名上标注为"纯数字账号登录"，但逻辑上是密码重置）。

不过该函数是 Server Action，需要先认证后才能调用（但认证机制本身也是基于 localStorage，存在被绕过可能）。

每个用户都可以无限制调用该接口获取/创建邀请码。虽然每次都只是查询或创建一条记录，但缺少速率限制可能被利用。

L-3: HTML 代理接口可能成为 XSS 来源

▶

问题描述：

/api/html-proxy 会将存储的 HTML 内容以 text/html 类型返回给浏览器。虽然有严格的白名单检查（仅允许 Supabase Storage 的公开 bucket + .html 文件），但如果 Storage 中的 HTML 文件被恶意篡改注入 XSS 代码，受害者访问时会执行攻击脚本。

建议对返回的 HTML 内容进行 CSP 或 script 标签过滤。

🔵
L-4: findAuthUserByEmail 逐页遍历所有用户（性能问题）
app/actions/auth.ts 低风险 性能
▶

问题描述：

在注册时遇到"邮箱已占用"错误时，会遍历所有 Auth 用户（每页200条，最多30页=6000用户）来找到对应用户。这在用户量大时性能很差。建议通过 Supabase Admin API 的直接用户查询替代。

三、逻辑漏洞

🟠
LOG-1: 游客阅读限制仅依赖 localStorage，无法防止真正滥用
高风险 业务逻辑
▶

游客（未登录用户）的阅读篇数记录在 localStorage 中，清除浏览器数据或使用隐身模式即可无限重置。这意味着任何人都可以通过不登录来绕过 3 篇免费额度限制。

这可能是产品设计决策（降低注册门槛），但应明确告知用户这个限制的绕过方式。

🟡
LOG-2: 邀请码可能重复生成
中风险 业务逻辑
▶

邀请码生成使用 8 位随机字符串（不含混淆字母如 0,O,1,I,l），理论上存在碰撞风险（约 22 亿分之一的碰撞概率）。如果发生碰撞，数据库 unique 约束会阻止插入，但用户会收到"邀请码创建失败"而非"邀请码已存在"的提示。

建议改用更长的邀请码或 UUID，并捕获 unique 约束冲突后重试生成。

< class="severity-icon si-medium">🟡
LOG-3: 会员过期后未自动降级
中风险 业务逻辑
▶

会员状态的判断逻辑是读取 memberships 表的 end_date 字段，但前端 membership-provider.tsx 和后端 API 可能在会员过期后仍返回旧状态（因为依赖缓存）。需要确认 memberships 表有定时任务或触发器在过期时更新 vip_tier。

建议在后端 membership status API 中实时查询 memberships 表的 end_date，而不仅仅依赖缓存。

🔵
LOG-4: pending_registrations 验证码可重复使用
低风险 业务逻辑
▶

验证码验证成功后虽然删除了 pending_registrations 记录，但如果用户在删除前重复提交（或网络重试导致多次调用 verifyEmailCode），可能创建多个重复的 Auth 用户。应该使用数据库事务或幂等性检查。
