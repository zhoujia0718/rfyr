三、安全问题

严重 Cookie 认证缺少关键安全属性 app/api/admin/login/route.ts
问题描述：管理员登录设置的 cookie 存在以下安全隐患：
  • httpOnly: false — Cookie 可被 JavaScript 读取，容易遭受 XSS 攻击窃取会话
  • sameSite: "lax" — 未设置为 strict，可能遭受 CSRF 攻击
  • 缺少 secure: true — 在 HTTPS 环境下Cookie不会被标记为安全
// 当前代码 (第62-75行)
response.cookies.set("admin-session-local", sessionPayload, {
  path: "/",
  maxAge: 60 * 60 * 24 * 7,
  sameSite: "lax",
  httpOnly: false,  // ❌ 应该为 true
})
建议修复：
response.cookies.set("admin-session-local", sessionPayload, {
  path: "/",
  maxAge: 60 * 60 * 24 * 7,
  sameSite: "strict",  // 防止 CSRF
  httpOnly: true,      // 防止 XSS 窃取
  secure: true,        // 仅 HTTPS 传输
})
严重 伪造 Token 认证机制存在安全风险 lib/server-auth-user.ts
问题描述：getUserIdFromBearer() 函数在检测到伪造 Token（pwd_/magic_ 前缀）时，直接信任 X-User-Id 请求头：
  • 攻击者可以构造任意伪造 Token + 任意 X-User-Id 伪装成其他用户
  • 没有验证该 X-User-Id 是否与伪造 Token 中的用户一致
  • URL 参数 ?uid= 同样可被伪造
// 当前代码 (第54-70行) - 存在伪造风险
// ── 伪造 token：信任 x-user-id header
const userIdHeader = request.headers.get("x-user-id")
if (userIdHeader?.trim()) {
  return userIdHeader.trim()  // ❌ 任何人都可以伪造
}

// ── 伪造 token：信任 ?uid= query 参数
const uidParam = url.searchParams.get("uid")
if (uidParam?.trim()) {
  return uidParam.trim()  // ❌ 攻击者可利用
}
风险：任何知道其他用户 ID 的攻击者都可以通过伪造请求头冒充该用户执行操作（如阅读记录、兑换会员等）。
高危 管理员页面缺少服务端权限校验 app/admin/page.tsx
问题描述：管理后台 /admin 页面完全依赖前端检查用户权限：
在 useEffect 中通过 localStorage 和 Supabase session 检查是否登录，但这是纯客户端校验：
  • 攻击者可以直接访问管理员页面 HTML（SSR/SSG 会输出完整内容）
  • 可调用管理后台 API（除非 API 本身有服务端校验）
  • 可查看其他用户的会员数据等敏感信息
// 当前代码仅在客户端检查 (第69-118行)
React.useEffect(() => {
  const loadData = async () => {
    let isAuthenticated = false
    // 1. 优先检查 localStorage
    const stored = localStorage.getItem('custom_auth')
    // ... 仅客户端检查 ...
    if (!isAuthenticated) {
      toast.error('请先登录')
      window.location.href = '/'  // 可被绕过
      return
    }
    // ... 加载数据 ...
  }
  loadData()
}, [])
建议：管理后台应该使用服务端渲染 + 中间件保护，确保未授权用户无法访问任何内容。
高危 会员状态 API 缺少签名验证 app/api/membership/status/route.ts
问题描述：/api/membership/status 使用 Service Role Key 直接查询数据库获取会员状态。当伪造 Token 配合伪造 X-User-Id 时，攻击者可以查询任意用户的会员状态信息。
高危 验证码存储在数据库中未加密 app/actions/auth.ts
问题描述：注册验证码以明文形式存储在 pending_registrations 表中：
  • 如果数据库被拖库，攻击者可以直接使用验证码注册账号
  • 建议：使用哈希存储验证码，或使用一次性哈希 Token
// 当前代码 (第391行)
const code = Math.floor(100000 + Math.random() * 900000).toString()
// 直接存入数据库
await supabaseAdmin.from('pending_registrations').upsert({
  email: trimmedEmail,
  code,  // ❌ 明文存储
  password,  // ❌ 密码也未加密
  ...
})
中危 调试接口在生产环境可能被意外调用 app/api/debug/email/route.ts
问题描述：虽然生产环境会返回 404，但调试接口依赖 process.env.NODE_ENV 判断。某些部署平台可能存在环境变量配置问题导致判断失效。

四、逻辑漏洞

高危 阅读限制前端可被绕过 hooks/use-reading-limit.ts
问题描述：游客的阅读篇数记录在 localStorage 中，这是完全不可信的客户端存储：
  • 用户可以清除 localStorage 重置阅读次数
  • 用户可以修改 localStorage 中的阅读篇数
  • 用户可以使用多个浏览器 profile 绕过限制
  • API 层虽然有校验，但仅针对已登录用户
// 当前代码 (第48-57行) - 仅限游客
if (!isUserLoggedIn) {
  // 游客：从 localStorage 恢复 (可被篡改)
  try {
    const raw = localStorage.getItem(STORAGE_KEY)
    const visited: string[] = raw ? JSON.parse(raw) : []
    setReadCount(visited.length)
  } catch {
    setReadCount(0)
  }
  return
}
建议：考虑对游客也使用设备指纹 + IP 的服务端追踪，或限制游客的阅读权限（如仅能在首页预览，详细的阅读需要登录）。
高危 兑换码次数限制可被绕过 lib/redeem.ts
问题描述：周卡兑换次数限制（免费1次 + 付费4次 = 最多5次）仅通过 weekly_purchase_count 字段在 user_profiles 表中追踪。如果用户：
  • 删除本地存储的 custom_auth 并注册新账号
  • 使用不同的邮箱重复注册
  • 可以无限制使用免费周卡
虽然 sendEmailVerificationCode 会在验证码阶段检查邮箱是否已存在，但用户可以使用不同的邮箱绕过。
中危 会员有效期计算存在时区问题 lib/redeem.ts
问题描述：会员有效期使用 new Date() 计算，基于服务器本地时区。兑换码过期检查也使用相同方式： 
// 第105行
if (redeemCodeData.status === "expired" || new Date(redeemCodeData.expires_at) < new Date()) {
  return { success: false, message: "兑换码已过期" }
}
  • 如果服务器时区与用户所在时区不同，可能出现显示不一致
  • 建议：使用 UTC 时间戳进行计算和比较
中危 邀请码生成可能产生碰撞 lib/referral.ts
问题描述：用户邀请码使用用户 UUID 前8位生成： 
// app/actions/auth.ts 第525行
const newUserCode = authData.user.id.replace(/-/g, '').slice(0, 8).toLowerCase()
虽然 UUID 碰撞概率极低，但如果两个用户的 UUID 前8位相同，会产生重复邀请码。数据库的 unique 约束可以防止插入失败，但用户体验不佳（看不到明确的错误）。
中危 会员顺延逻辑可能导致超长有效期 lib/redeem.ts
问题描述：兑换码兑换时，如果用户已有同类有效会员，会顺延日期： 
// 第153-162行
if (existingMembership) {
  // 顺延
  const existingEnd = new Date(existingMembership.end_date)
  const newEnd = new Date(existingEnd)
  newEnd.setDate(newEnd.getDate() + days)
  // ... 更新 end_date
}
  • 理论上用户可以无限续期（每次顺延7天或365天）
  • 建议：设置最大有效期限制（如年卡最多3年）
中危 验证码有效期10分钟可能过短 app/actions/auth.ts
问题描述：邮箱验证码有效期设为10分钟 (10 * 60 * 1000)，对于某些邮箱服务（特别是企业邮箱或163邮箱）可能较难及时送达。建议考虑增加至15分钟或提供重新发送机制。

五、代码质量问题

低 死代码遗留 lib/payments.ts
问题描述：lib/payments.ts 文件仍然存在但所有引用已被删除。这是微信支付相关的代码，在移除支付功能后未清理。
低 已删除页面的路由可能仍被引用 app/admin/verify-payments/page.tsx
问题描述：app/admin/verify-payments/page.tsx 已从文件系统删除，但 Git 显示为 "D" 状态。检查是否有其他代码引用此路由（如导航链接）。
低 console.log 调试输出未清理 多处
问题描述：多处代码包含调试用的 console.log 语句，在生产环境中可能泄露敏感信息：
  • lib/server-auth-user.ts — 打印 authorization header 和 token
  • app/api/referral/code/route.ts — 打印 authorization 和 cookie
  • app/notes/[slug]/page.tsx — 打印邀请码信息
低 魔法数字未定义常量 多处
问题描述：多处使用硬编码数字，缺乏语义化的常量定义：
  • FREE_READ_COUNT = 3 — 硬编码在 hook 中
  • WEEKLY_LIMIT = 10 — 硬编码在多个页面
  • MAX_WEEKLY_TOTAL = 4 — 在 lib/redeem.ts 中
建议：统一在 lib/membership.ts 中定义所有限制常量。
低 错误处理不一致 多处 API
问题描述：不同 API 路由的错误处理方式不一致：
  • 有些返回 status: 500 带详细错误信息
  • 有些只返回 status: 400
  • 数据库错误被直接暴露给前端可能泄露数据库结构信息
