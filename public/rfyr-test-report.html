一、认证系统问题

4 个问题
严重 伪造 Token 可被未授权用户利用 lib/server-auth-user.ts

系统支持"伪造 Token"机制，允许客户端自行生成 token（如 fake_userId_expiry_signature）。当 HMAC_SECRET 环境变量未配置时，伪造 token 完全不验证签名，任何人都可以构造任意用户 ID 的请求。

代码逻辑：

// lib/server-auth-user.ts 第20-24行 function verifyFakeTokenSignature(token: string, userId: string): boolean { if (!HMAC_SECRET) { // 未配置 HMAC_SECRET 时，降级为不验证签名（安全风险） console.warn("[Auth] 警告: 未配置 HMAC_SECRET，伪造 token 未签名验证") return true // ⚠️ 任何人可伪造任意用户身份 } ... // 旧格式伪造 token（pwd_/magic_ 前缀），不验证签名 if (token.startsWith("pwd_") || token.startsWith("magic_")) { console.warn("[Auth] 警告: 使用旧格式伪造 token，无签名验证") return true // ⚠️ 任何人可伪造任意用户身份 } }
影响：攻击者可伪造任意用户身份访问付费内容、兑换码、邀请系统等。

修复建议

  • 必须配置 HMAC_SECRET 环境变量
  • 移除旧格式 token（pwd_/magic_）的向后兼容
  • 添加启动检查：未配置 HMAC_SECRET 时禁止启动
  • 考虑迁移到标准 Supabase JWT 认证
高危 x-user-id Header 信任问题 app/api/membership/status/route.ts

会员状态 API 直接信任 x-user-id header，即使没有有效的认证 token。这个 header 可以在客户端随意设置，允许用户查看任意用户的会员状态。

// app/api/membership/status/route.ts 第15-22行 let userId: string | null = null if (userIdHeader?.trim()) { userId = userIdHeader.trim() // ⚠️ 直接信任客户端提供的 header } else if (authHeader?.toLowerCase().startsWith("bearer ")) { // 只有在没有 x-user-id 时才验证真实的 Supabase token ... }
影响：用户可查看任意用户的会员等级信息（虽不能直接修改，但泄露用户隐私）。

修复建议

  • x-user-id 必须配合有效的签名 token 使用
  • 使用 Supabase 真实 token 验证用户身份
  • 在 lib/server-auth-user.ts 中统一认证逻辑
高危 邀请码验证无身份绑定 app/api/referral/create-codes/route.ts

普通用户可以调用生成兑换码 API（/api/referral/create-codes）。虽然兑换码有"不能使用自己生成的"限制，但攻击者可以批量生成兑换码发给其他用户使用，造成资源滥用。

影响：非管理员用户可以批量生成兑换码，可能被用于灰产交易。

修复建议

  • 此 API 应限制为管理员权限
  • 添加 requireAdmin() 认证检查
  • 记录兑换码生成日志便于审计
中危 旧版 admin-session Cookie 安全性不足 lib/server-admin-auth.ts

旧版 admin-session cookie 直接存储 userId，没有设置 httpOnly 和 secure 属性。且在 proxy.ts 中，只要存在此 cookie 就认为已认证，没有验证过期时间。

// proxy.ts 第48-54行 if (!isAuthenticated) { const adminSession = request.cookies.get("admin-session") if (adminSession?.value && adminSession.value.length > 0) { isAuthenticated = true // ⚠️ 没有过期时间检查 } }
影响：如果旧 cookie 被窃取，攻击者可以永久使用（无过期时间）。

修复建议

  • 移除旧版 cookie 的向后兼容
  • 确保所有认证都使用带过期时间的新版 cookie

二、API 安全问题

4 个问题
严重 IP 限流使用内存存储，服务器重启后失效 app/api/referral/validate/route.ts

邀请码验证 API 的 IP 限流使用 Map 存储在内存中。当服务器重启或使用多实例部署时，限流记录会丢失，攻击者可重新发起暴力破解。

// app/api/referral/validate/route.ts 第14行 const ipMap = new Map<string, { count: number; resetAt: number }>() // ⚠️ 内存存储，服务器重启后丢失
影响：暴力破解邀请码的风险增加。

修复建议

  • 使用 Redis 等外部存储保存限流数据
  • 或在 Supabase 中记录验证尝试次数
高危 阅读限制记录使用非原子操作 lib/reading-limit.ts

阅读次数记录使用"先查询后更新"模式，虽然代码注释说是"原子 upsert"，但实际上是两次数据库操作，存在并发竞态条件。

// lib/reading-limit.ts 第75-103行 // 原子 upsert：先查再写，防止并发问题 ← 注释说的是原子，但代码不是 const { data: profile } = await supabaseAdmin // 第一次查询 .from("user_profiles") .select("notes_read_count, notes_read_ids") ... // ... 计算 newCount ... await supabaseAdmin // 第二次更新 .from("user_profiles") .update({ notes_read_count: newCount }) ...
影响：在并发请求时，同一用户可能超出阅读限制。

修复建议

  • 使用 Supabase RPC 函数实现真正的原子操作
  • 或使用数据库行级锁
中危 错误信息泄露实现细节 多处

API 错误响应中包含数据库错误详情，如 error.code、error.message 等，可能被攻击者利用获取系统信息。

影响：攻击者可了解数据库类型、表结构等内部信息。

修复建议

  • 将数据库错误转换为通用错误消息
  • 在生产环境记录详细错误日志，但返回给客户端的是脱敏信息
低危 邀请码验证未验证 token 签名 app/api/referral/validate/route.ts

邀请码验证 API 没有进行用户身份验证，任何人都可以验证任意邀请码（但有 IP 限流保护）。

影响：较低，已有限流保护。但可能泄露邀请码是否有效的信息。

修复建议

  • 考虑要求用户先登录才能验证邀请码
  • 或限制验证次数（不仅是 IP 级别，还包括用户级别）

三、会员和付费墙问题

3 个问题
高危 会员状态可被客户端伪造 components/membership-provider.tsx

会员 Provider 在获取会员状态时，将 vip_tier 存储在 localStorage 中。理论上攻击者可以通过修改 localStorage 伪造会员状态（虽然服务端 API 会再次验证）。

// components/membership-provider.tsx 第88-92行 const fetchSilent = useCallback(async () => { const stored = getMembershipFromStorage() if (stored && isMembershipValid(stored)) setMembership(stored) const info = await fetchMembershipFromBackend() setMembership(info) }, ...)
影响：如果服务端 API 存在漏洞（如信任 x-user-id header），攻击者可能绕过付费墙。

修复建议

  • 不依赖 localStorage 存储敏感状态
  • 所有权限检查必须通过服务端 API
  • 客户端只做显示，不做权限判断
中危 付费墙模糊效果可被 CSS 移除 components/paywall.tsx

付费墙使用 CSS blur() 和 pointer-events: none 隐藏内容，但这些样式可以在浏览器开发者工具中轻松移除。

// components/paywall.tsx 第119-121行 <div class="blur-sm pointer-events-none select-none opacity-50"> {children} // ⚠️ 内容仍然在 DOM 中，只是视觉隐藏 </div>
影响：技术用户可以绕过付费墙查看预览内容（虽然不能获取完整内容）。

修复建议

  • 不要将完整内容发送到客户端
  • 服务端只返回允许预览的部分内容
  • 使用 React 的条件渲染，限制内容根本不出现在 DOM 中
中危 月卡每日限制未按日期重置 app/api/articles/[id]/route.ts

代码注释说月卡用户"每日 8 篇"，但实现中只记录了总阅读次数 notes_read_count，没有按日期区分。实际效果是月卡用户终身只能看 8 篇，而不是每天 8 篇。

影响：月卡会员权益被错误实现，可能导致用户投诉。

修复建议

  • 如果设计是"每天 8 篇"，需要添加日期字段记录
  • 如果设计是"总共 8 篇"，更新用户界面说明
  • 在数据库添加 daily_read_count 和 last_read_date 字段

四、邀请系统问题

2 个问题
中危 邀请码格式验证过于宽松 app/api/referral/validate/route.ts

邀请码验证只检查长度不超过 16 位，没有检查格式是否正确。合法的邀请码应该符合特定格式（如 8 位小写字母数字），但系统会查询数据库中不存在的任意字符串。

影响：增加数据库查询压力，可能泄露数据库中是否存在特定邀请码。

修复建议

  • 添加正则表达式格式验证
  • 拒绝明显不符合格式的邀请码
低危 邀请码大小写处理不一致 多处

创建邀请码时使用小写存储（code.toLowerCase()），但登录表单允许用户输入大写并自动转换为小写。这种不一致可能导致混淆。

影响：用户可能认为邀请码区分大小写，造成使用困惑。

修复建议

  • 统一邀请码格式为大写或小写
  • 在 UI 上明确显示邀请码格式要求

五、业务逻辑问题

2 个问题
中危 年卡顺延逻辑计算错误 lib/redeem.ts

年卡顺延时使用 yearsToAdd = daysToAdd / 365 然后 setFullYear(getFullYear() + yearsToAdd)。当 daysToAdd = 365 时，yearsToAdd = 1 是正确的，但如果有多次顺延，累积的浮点数误差可能导致日期不准确。

// lib/redeem.ts 第60-68行 if (membershipType === "yearly") { const yearsToAdd = Math.min(daysToAdd / 365, MAX_MEMBERSHIP_YEARLY_EXTENSIONS) newEnd.setFullYear(newEnd.getFullYear() + yearsToAdd) // ⚠️ 浮点数计算可能不精确 }
影响：年卡多次顺延时，有效期计算可能不准确。

修复建议

  • 使用整数计算：setFullYear(getFullYear() + 1)
  • 添加单元测试验证边界情况
低危 用户免费月卡限制检查逻辑 lib/redeem.ts

免费月卡限制检查依赖 source !== "purchase" 判断，但数据库中 source 字段可能为 null，导致 null !== "purchase" 为 true，被误判为免费兑换。

// lib/redeem.ts 第253行 const isFreeUse = !redeemCodeData.source // source 为 null/undefined 时为 true
影响：如果兑换码的 source 字段为 null，可能绕过免费月卡限制。

修复建议

  • 明确 source 字段的取值范围
  • 使用更严格的类型检查
