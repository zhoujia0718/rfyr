1️⃣ 系统概览

阅读限制系统控制免费和低级会员用户的访问配额，防止内容被大量爬取，同时激励用户升级会员。系统追踪两类用户的阅读行为：已登录用户（服务端数据库持久化）和未登录游客（客户端 localStorage 防篡改 + 服务端 guest_reads 表兜底）。

核心数据表

📄 user_profiles 表

  • notes_read_count — 终身已读篇数（仅 notes 分类）
  • notes_read_ids — 已读文章 ID 数组（防重复计数）
  • daily_read_count — 当日已读篇数（每日重置）
  • last_read_date — 最后阅读日期（判断是否跨天）
  • bonus_read_count — 终身邀请奖励次数（非会员）
  • bonus_daily_count — 每日邀请奖励次数（会员）
  • bonus_daily_reset_date — 每日奖励重置日期（CST）

📄 guest_reads 表

  • guest_id — 游客身份标识（IP+UA 哈希）
  • read_ids — 已读文章 ID 数组
  • read_count — 已读篇数
  • expires_at — 过期时间（30天后自动清理）
  • 自动清理函数：cleanup_expired_guest_reads()

📄 reading_settings 表

  • id — 主键（固定为 'global'）
  • guest_read_limit — 游客每日阅读上限（默认 3）
  • monthly_daily_limit — 月卡每日上限（默认 8）
  • referral_bonus_count — 每次邀请奖励次数（默认 2）

📄 referrals 表

  • referrer_id — 邀请人 ID
  • referee_id — 被邀请人 ID
  • referrer_article_id — 来源文章 ID（可选）
  • 唯一约束：每对 (referrer, referee) 仅一条
关键设计决策：配额重置基于中国标准时间（CST，UTC+8），使用自定义 toLocalDateString() 函数将 Date.getTime() + 8h 后再取 toISOString().split('T')[0]，解决 new Date().toISOString() 返回 UTC 0 点导致跨日判断错误的问题。

recordVisit 核心逻辑伪代码

// lib/reading-limit.ts - recordVisit()
async function recordVisit(userId, articleId):
    today = toLocalDateString()  // CST 今日

    // Step 1: 查询当前已读状态
    profile = SELECT FROM user_profiles WHERE id = userId

    existingIds = profile.notes_read_ids ?? []
    alreadyRead = existingIds.includes(articleId)
    lastReadDate = profile.last_read_date.split('T')[0]
    shouldResetDaily = lastReadDate !== today

    // Step 2: 构建更新值
    if (!alreadyRead):
        newIds = [...existingIds, articleId]
        newCount = newCount + 1
        newDailyCount = (shouldResetDaily ? 0 : newDailyCount) + 1
    else:
        newDailyCount = shouldResetDaily ? 1 : newDailyCount

    // Step 3: UPDATE（注意：非原子操作）
    UPDATE user_profiles SET
        notes_read_count = newCount,
        notes_read_ids = newIds,
        daily_read_count = (shouldResetDaily ? 1 : newDailyCount),
        last_read_date = today
    WHERE id = userId
⚠️ 关键缺陷：非原子操作。Step 1 查询和 Step 3 更新之间存在时间窗口，并发请求可能导致计数不准确。详见下方问题 #1。

8️⃣ 问题清单与解决方案

经过对模块四所有文件的详细代码审查，共识别出 10 个问题，涵盖安全性、数据一致性、性能、逻辑缺陷等维度：

🔴 P1 非原子操作 🔴 P2 游客 localStorage 可篡改 🟠 P3 双重存储不一致 🟠 P4 并发重复记录 🟡 P5 游客配额无法服务端强制 🟡 P6 多实例缓存不同步 🟡 P7 游客计数未区分来源 🔵 P8 游客 API 端点缺失 🔵 P9 Cron Job 未配置 🔵 P10 年度会员每日限制未使用

🔴 问题 1：recordVisit 非原子操作导致计数不准确 严重 P1 - 安全性/数据一致性

位置：lib/reading-limit.ts 第 92-160 行 recordVisit() 函数

问题描述：函数先执行 SELECT 查询当前状态，再执行 UPDATE 更新新状态。这两步之间存在时间窗口，当用户在两台设备同时阅读、或短时间内快速打开多篇文章时，多个请求可能同时读到相同的旧值，导致计数比实际少 1-多条。

// 当前代码（非原子）：
profile = SELECT FROM user_profiles WHERE id = userId  // 时刻 T1
  // ⚠️ 并发请求也在此时 SELECT，得到相同数据
newCount = oldCount + 1
UPDATE user_profiles SET notes_read_count = newCount  // 时刻 T2
✅ 解决方案：使用 Supabase 原子操作或数据库事务

方案 A（推荐）：原子 UPDATE + RETURNING
将查询和更新合并为一个原子 UPDATE 操作：

🔴 问题 2：游客 localStorage 计数完全可被用户篡改 严重 P2 - 安全性

🟠 问题 3：游客已读状态存在双重存储，行为不一致 高 P3 - 数据一致性

位置：hooks/use-reading-limit.ts 全文 + supabase/migrations/guest_reads.sql

问题描述：系统设计了两套游客追踪机制，但只用了其中一套：

  • localStorage（已使用）：rfyr_visited_notes，客户端独立维护
  • Supabase guest_reads（已建表但未接入）：服务端持久化，IP+UA 哈希标识，30天过期

🟠 问题 4：前端 recordVisit 中存在双重查询浪费与潜在竞态 高 P4 - 性能/逻辑

位置：hooks/use-reading-limit.ts 第 183-206 行

问题描述：已登录用户的 recordVisit 流程中：

  1. 组件挂载时已经调用过一次 GET /api/reading-limit 获取初始数据（第 71-98 行）
  2. 写入前又调用了一次 GET /api/reading-limit 比对 readIds（第 191-206 行）
  3. 确认未记录后再调用 POST /api/reading-limit（第 223-227 行）

一次阅读记录最多产生 3 次网络请求，且第 2 步的"预查询"与服务端 recordVisit 内部的 SELECT 形成了两次查询。此外，第 2 步 GET 和第 3 步 POST 之间存在竞态：两次请求之间若有其他标签页同时写入，会导致重复。

🟡 问题 6：多实例部署时 settingsCache 不同步 高 P6 - 性能/一致性

位置：lib/reading-settings.ts 第 27-28 行

问题描述：settingsCache 是模块级内存变量。当 Next.js 应用部署在多个实例（多个 Node.js 进程）上时：

  • 实例 A 的管理员 PUT 新设置 → 实例 A 缓存清空，实例 B/C 缓存仍是旧值
  • 60 秒后旧实例缓存过期才读取新值（最大延迟 60 秒）
  • 更严重的是：若 Supabase 被外部工具修改（如 Supabase Dashboard SQL Editor），所有实例的内存缓存都不会更新
✅ 解决方案：移除内存缓存，依赖数据库缓存

Next.js Vercel/Vercel 部署时每个请求是独立的冷启动环境，内存缓存意义不大。直接每次从数据库读取（Supabase 有内置连接池和缓存），或者使用 revalidateTag 让 Next.js 的 Data Cache 机制处理：
// Next.js 15 App Router 方案
export async function GET() {
    // 每次从数据库读取，利用 Next.js Data Cache
    const settings = await unstable_cache(
        () => getSettingsFromDb(),
        ['reading-settings'],
        { revalidate: 60, tags: ['reading-settings'] }
    )()
    return NextResponse.json(settings)
}

// PUT 时清除缓存
export async function PUT() {
    await db.update(...)
    revalidateTag('reading-settings')  // 清除所有实例缓存
}
