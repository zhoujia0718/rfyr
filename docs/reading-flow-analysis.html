6阅读计数与邀请奖励 — 数据流全图

写入路径
客户端 mount useReadingLimit → fetch GET /api/reading-limit → 读数据库
客户端 mount useDailyQuotaCheck → fetch GET /api/reading-limit + GET /api/reading-settings
用户点击文章（URL 导航） GET /api/articles/[id] → recordUserRead → 写 daily_read_count + notes_read_count ⚠️ 月卡超限后仍写入，bug
文章页渲染后（effect） 客户端 recordVisit → POST /api/reading-limit → recordVisit → 写数据库 ⚠️ 双重写入：服务端已写一次，客户端再写一次
好友注册成功 createReferral → 写 bonus_read_count（非会员）或 bonus_daily_count（会员）
数据库字段含义
字段 作用域 写入时机 重置
notes_read_count 终身 每次阅读新文章 永不
notes_read_ids 终身 每次阅读新文章 永不（防重复）
daily_read_count 每日 每次阅读新文章 次日北京时间零点（按 last_read_date 判断）

bonus_daily_count 邀请奖励·每日 月卡/季卡/年卡邀请成功 +N 次日北京时间零点（按 bonus_daily_reset_date 判断）

last_read_date 辅助字段 每次阅读 —（用于判断是否跨天）
bonus_daily_count 每日重置条件： bonus_daily_reset_date !== today 时， 写入 bonus_daily_count = referral_bonus_count（从 0 开始累加）。 用户无邀请时该值为历史遗留值，需手动 SQL 清零。

7月卡每日限额 — 计算公式

effectiveDailyLimit 如何得出
effectiveDailyLimit = monthly_daily_limit + dailyBonusCount

其中：
  monthly_daily_limit   = 来自 reading_settings 表（默认 8）
  dailyBonusCount      = bonus_daily_count（当 reset_date === today 时）
                    = 今日邀请成功人数 × referral_bonus_count
                    = 今日被邀请人注册数（referrals 表）

例：月卡用户，今日邀请 1 人，referral_bonus_count = 2
→ effectiveDailyLimit = 8 + 2 = 10 篇/天
前端 vs 服务端一致性： 前端 useDailyQuotaCheck 用 dailyBonusCount（来自 GET /api/reading-limit 的 dailyBonusCount 字段）， 服务端用同样的 bonus_daily_count 数据库字段。 两者来源相同，正常情况下数值一致。

8完整用户路径 — 月卡用户的一天

月卡用户：每日阅读 + 邀请流程（北京时间为准）
00:00 北京时间 bonus_daily_reset_date 不等于今天 → bonus_daily_count = 0（理论上）；last_read_date 不等于今天 → daily_read_count = 0（按需重置）
上午 阅读第 1 篇 → recordUserRead：daily_read_count = 1，notes_read_count = 1
上午 阅读第 2、3、...N 篇 → daily_read_count 累加
阅读第 8 篇时 服务端：daily_read_count = 8，effectiveDailyLimit = 10 → 放行（未超限）
邀请 1 位好友注册 createReferral：bonus_daily_count = 0 + 2 = 2，bonus_daily_reset_date = 今天 effectiveDailyLimit 从 8 → 10
阅读第 9 篇 recordUserRead：daily_read_count = 9，effectiveDailyLimit = 10 → 放行
阅读第 10 篇 recordUserRead：daily_read_count = 10，effectiveDailyLimit = 10 → 刚好达到上限，放行
尝试阅读第 11 篇 侧边栏守卫：daily_read_count(10) >= effectiveDailyLimit(10) → 弹窗，不跳转
直接访问 URL 服务端：recordUserRead → daily_read_count = 11 → 返回 403 DAILY_LIMIT_EXCEEDED ⚠️ bug：11 >= 10，已超限但仍写入
X 掉弹窗 留在原文章页，可以继续阅读当前已加载的文章
明天 00:00 bonus_daily_count 归零（bonus_daily_reset_date !== today）；daily_read_count 归零（last_read_date !== today）

9发现的问题汇总

  • Bug 1：服务端超限后仍写入 daily_read_count
    recordUserRead 在判断超限之前执行，导致月卡用户被 403 后计数仍 +1。
    影响：反复尝试访问被拦截的文章，daily_read_count 不断累加，最终可能错误拦截本可读的文章。
    ✅ 已修复（2026-04-18）：GET 路由改为先 dry-run 推算是否超限，只在未超限时才写入。超限直接返回 403。
  • Bug 2：月卡用户可见年卡文章内容（仅弹窗提示）
    服务端对年卡专属文章返回 403 但仍附带 content，article page 渲染后弹窗可关闭。
    影响：月卡用户可绕过年卡专属内容限制，关闭弹窗后继续阅读年卡文章。
    ✅ 已修复（2026-04-18）：年卡专属文章返回 403 时移除 content 字段，弹窗关掉后内容区为空。
  • Bug 3：侧边栏守卫首次渲染可能错误拦截
    effectiveDailyLimit 初始值为 0，用户点击早于 quota API 响应时，0 >= 0 为 true。
    影响：月卡用户首次访问时，极其短暂（ms 级）的错误拦截。正常使用时基本不可感知。
    ✅ 已修复（2026-04-18）：新增 quotaLoaded 状态，守卫在数据未加载完时直接放行。
  • Bug 4：客户端重复写入阅读记录
    服务端 recordUserRead 已写一次 DB，客户端 recordVisit effect 再写一次（双重写入）。
    影响：两次写入结果相同（+1），数据正确但不必要，增加 DB 负载。
    ✅ 已修复（2026-04-18）：客户端 recordVisit 在调用 API 前先 fetch 一次 readIds，已包含该文章则跳过写入。
  • Bug 5：bonus_daily_count 历史遗留值
    2026-04-18 之前创建的用户，bonus_daily_count 可能为非零历史值，不会自动重置。
    影响：用户今日没有邀请，但 bonus_daily_count > 0，导致 effectiveDailyLimit 偏高。
    ⚠️ 待执行（2026-04-18）：迁移脚本 20260418_bonus_daily_reset.sql 已创建，API 已加兜底归零逻辑。用户需在 Supabase SQL Editor 执行迁移脚本：
    ALTER TABLE user_profiles ADD COLUMN IF NOT EXISTS bonus_daily_reset_date DATE DEFAULT '1970-01-01'; UPDATE user_profiles SET bonus_daily_count = 0, bonus_daily_reset_date = '2026-04-18' WHERE bonus_daily_count > 0;
  • Bug 6：侧边栏守卫无「已读不计数」逻辑
    服务端有 alreadyRead 判断（已读过的文章不增加计数），但侧边栏前端守卫没有。
    影响：如果 daily_read_count 已被错误累加（Bug 1），侧边栏会拦截已读过的文章；服务端则因为 alreadyRead 不计数的双重保护而放行。
    ✅ 已修复（2026-04-18）：useDailyQuotaCheck 返回 readIds，守卫里用 Set 查询，已读过的文章跳过前端拦截。
