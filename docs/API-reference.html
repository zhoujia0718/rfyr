一、认证机制说明

用户认证（App 层）

用户认证采用复合机制，存在两种 Token 格式：

Token 类型格式验证方式安全性
真实 Supabase Token eyJhbGci... Supabase getUser() 验证 ✅ 安全
伪造 Token（fake） pwd_{userId} / magic_{userId} 信任 X-User-Id Header ⚠ 依赖 HMAC_SECRET 是否配置
⚠ HMAC 签名验证： 伪造 Token 需要配合 HMAC 签名（X-HMAC-Signature Header）才能通过验证。 签名 = HMAC-SHA256(token, HMAC_SECRET) 的前 16 位十六进制。 启用方式：在 .env.local 中设置 HMAC_SECRET=your_secret_key。 若未配置，伪造 Token 可被任意构造。
✅ 已修复（2026-04-18）： ?uid= URL 参数已被移除，不再作为认证来源。 验证码存储改为 SHA-256 哈希（salt: rfyr_salt_v1）。

管理员认证（Admin 层）

管理员使用 Cookie 认证（admin-session-local），由 proxy.ts Middleware 强制检查。

Cookie 名格式过期时间安全属性
admin-session-local JSON（含 userId + loginTime） 7 天 ✅ httpOnly: true, sameSite: strict
admin-session 纯字符串 userId（旧版） 兜底兼容 ⚠ 仅向后兼容

二、新增 API（2026-04-18）

GET /api/reading-settings 🆕 新增

获取当前阅读限制配置（游客限额、月卡每日限额、邀请奖励次数）。允许任何人读取。

Request

GET /api/reading-settings Authorization: (可选) X-User-Id: (可选)

Response 200

{ "guest_read_limit": 3, "monthly_daily_limit": 8, "referral_bonus_count": 2 }

数据库表

reading_settings ├── id (TEXT, PK) = 'global' ├── guest_read_limit (INTEGER) -- 游客阅读上限 ├── monthly_daily_limit (INTEGER) -- 月卡每日上限 ├── referral_bonus_count (INTEGER) -- 每次邀请奖励次数 ├── created_at / updated_at (TIMESTAMPTZ)
服务端会清除 1 分钟缓存确保返回最新值。客户端有 5 分钟 localStorage 缓存。
PUT /api/reading-settings 🆕 新增

更新阅读限制配置（仅管理员可用）。会清除服务端缓存。

⚠ 认证：需要有效的 admin-session-local 或 admin-session Cookie。

Request

PUT /api/reading-settings Content-Type: application/json Cookie: admin-session-local=... { "guest_read_limit": 3, "monthly_daily_limit": 10, "referral_bonus_count": 3 }

Response 200

{ "success": true, "settings": { "guest_read_limit": 3, "monthly_daily_limit": 10, "referral_bonus_count": 3 } }
// Response 401 未授权 { "error": "未授权" }
GET /api/referral/stats 🆕 新增

获取当前用户的邀请统计数据（邀请人数、bonus 次数、会员状态）。需要用户认证。

Request

GET /api/referral/stats Authorization: Bearer {token} X-User-Id: {userId}

Response 200

{ "referralCount": 3, "bonusReadCount": 4, "bonusDailyCount": 6, "membershipType": "monthly", "referrerCode": "a1b2c3d4" }

返回字段说明

字段含义
referralCount累计邀请成功人数
bonusReadCount非会员邀请奖励（终身有效），来源：bonus_read_count
bonusDailyCount会员邀请奖励（每日重置），来源：bonus_daily_count
membershipType当前会员类型（none / monthly / yearly）
referrerCode用户的邀请码（无则自动创建）
GET /api/admin/me 🆕 新增

检查当前用户的管理员认证状态。无需认证（本身即检查认证状态）。前端判断是否已登录管理员账号。

Request

GET /api/admin/me Cookie: admin-session-local=...

Response 200（已登录）

{ "authenticated": true, "adminId": "uuid...", "email": "admin@example.com", "message": "已登录" }
// Response 401 未登录 { "authenticated": false, "message": "未登录管理员账号" }
GET /api/dev/login 🆕 新增

开发环境快捷登录。从 users 表取第一个用户作为测试用户，快速切换不同会员状态进行测试。

⚠ 安全：仅在 NODE_ENV !== 'production' 时可用。生产环境返回 {"ok": false, "error": "Not available in production"}。

Request

GET /api/dev/login

Response 200

{ "ok": true, "userId": "uuid...", "tier": "none", "message": "测试用户登录成功" }

三、重构 API

GET /api/articles/[id] 🔧 重构

获取单篇文章内容。核心变更：新增服务端强制阅读限制守卫，先 dry-run 推算是否超限，再决定是否写入 DB。月卡用户返回 403 时不再附带 content（防止绕过）。

权限判断流程（服务端）

1. 游客访问 access_level === "free" → 200 返回内容 access_level !== "free" → 401 REQUIRE_LOGIN 2. 已登录用户 article.access_level === "yearly" && !isYearly → 403 YEARLY_REQUIRED（无 content） 3. 月卡用户（含年卡用户） isMonthly || isYearly → dry-run: projectedDailyCount = dailyReadCount + 1 → projectedDailyCount > effectiveDailyLimit → 403 DAILY_LIMIT_EXCEEDED（不写入） 4. 非会员 !isMonthly && !isYearly → !alreadyRead && readCount >= maxLimit → 403 LIMIT_EXCEEDED 5. 通过全部检查 → recordUserRead（仅写入，不重复） → 200 返回文章内容

effectiveDailyLimit 计算

effectiveDailyLimit = monthly_daily_limit + dailyBonusCount 其中： monthly_daily_limit = reading_settings.monthly_daily_limit（默认 8，可后台配置） dailyBonusCount = bonus_daily_count（今日邀请成功人数 × referral_bonus_count） bonus_daily_count 重置条件：bonus_daily_reset_date !== today 时归零

服务端会员判断顺序

1. 优先查 memberships 表（status=active, end_date > now） 2. fallback 到 users.vip_tier 3. 支持 membership_type 多种写法：monthly / monthly_vip / monthly_vip_v2（统一 toLowerCase 后判断）
✅ 已修复（2026-04-18）： 月卡超限后不再写入 DB（dry-run 机制）；年卡专属文章返回 403 时 content 置空； 双重写入问题已解决（先检查 readIds 是否已包含再写入）。
GET /api/membership/status 未变

获取当前用户会员状态。认证方式同其他 API（Bearer token + X-User-Id Header）。

Request

GET /api/membership/status Authorization: Bearer {token} X-User-Id: {userId}

Response 200

{ "vip_tier": "none" | "monthly" | "yearly", "membership": { "type": "monthly_vip", "isActive": true, "endDate": "2026-05-18" } }
⚠ 认证说明：此 API 依赖 X-User-Id Header。 若 HMAC_SECRET 未配置，伪造 token + 伪造 X-User-Id 可冒充任意用户。
