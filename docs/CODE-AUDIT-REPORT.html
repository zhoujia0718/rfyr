项目概览

日富一日 (RFYR) 是基于 Next.js 16 (App Router) + React 19 + TypeScript + Supabase 的付费内容订阅平台，面向 A 股短线交易投资者群体。

核心功能模块：

  • 会员系统：月卡 / 年卡订阅，支持兑换码激活
  • 邀请关系：推荐链接 + 阅读奖励（非会员终身 / 会员每日）
  • 阅读限制：游客 / 非会员 / 月卡每日 / 年卡无限制，四级分层
  • 内容管理：文章 / 笔记 / 大佬合集 / 个股挖掘，支持 PDF / HTML 上传
  • 管理后台：文章编辑、兑换码管理、会员管理、阅读设置

技术栈：Next.js 16 · React 19 · TypeScript · Supabase (PostgreSQL + Auth + Storage + RLS) · Tailwind CSS · Radix UI · Zod

pending_registrations 表密码明文存储
app/actions/auth.ts:465  |  V-H-05 修复了哈希存储但密码仍明文

密码以明文写入 pending_registrations 表。注释声称"Supabase Auth 内部负责哈希存储"，但 pending_registrations 是自建表，不在 Supabase Auth 保护范围内。若邮件发送失败导致注册流程中断、或数据库被攻破，密码直接以明文泄露。

// app/actions/auth.ts line 465 const { error: upsertError } = await supabaseAdmin .from("pending_registrations") .upsert({ email: trimmedEmail, username: trimmedName, password, // ← 明文密码在此表停留 code: codeHash, expires_at: expiresAt, ... })

解决方案

不应在服务端明文存储密码。验证码验证通过后，应直接从用户输入（body）传递给 Supabase Auth 的 createUser，而不在 pending 表中暂存密码。如果必须暂存，应使用 bcrypt / argon2 哈希后再存，但这样又引入了解密问题——建议直接取消 pending 表存密码的机制。

// 正确做法：verifyEmailCode 中，验证通过后 // 直接从 pending 表读取的应该是 codeHash，不是 password // 密码应在 verifyEmailCode 调用时由客户端重新提交，或使用临时 token 机制 // 示例：verifyEmailCode 改为接收临时 token，验证后直接 createUser export async function verifyEmailCode( email: string, code: string, tempToken: string // 含密码哈希的临时令牌，由 sendEmail 时返回给客户端 ): Promise<...> { ... }

fakeToken 生成长度两处不一致（auth.ts vs server-auth-user.ts）
app/actions/auth.ts:661  |  lib/server-auth-user.ts:88

auth.ts（verifyEmailCode）生成的 token 截断为 16 字符，但 server-auth-user.ts（generateFakeToken）返回完整 64 字符。验证方 verifyFakeTokenSignature 期望 64 字符签名（line 39: /^[0-9a-f]{64}$/i），导致通过邮箱注册的用户无法通过服务端认证。

// lib/server-auth-user.ts:88 —— generateFakeToken（正确，完整64字符） function createHmacSignature(data: string, secret: string): string { const { createHmac } = require("crypto") return createHmac("sha256", key).update(msg).digest("hex") // ← 不截断 } // app/actions/auth.ts:658-661 —— verifyEmailCode（错误，截断16字符） const signature = createHmac('sha256', hmacSecret) .update(`${authData.user.id}_${expiresAt}`) .digest('hex') .slice(0, 16) // ← 截断！导致验证方校验失败 fakeToken = `fake_${authData.user.id}_${expiresAt}_${signature}`

解决方案

统一两处 fake token 生成逻辑：都使用 lib/server-auth-user.ts 中的 generateFakeToken 函数，或将 server-auth-user.ts 的签名逻辑抽取为共享工具函数。

// app/actions/auth.ts:650-665 —— 修改后 const hmacSecret = process.env.HMAC_SECRET let fakeToken: string | undefined if (hmacSecret) { // 直接复用 server-auth-user.ts 的 generateFakeToken const { generateFakeToken } = require('@/lib/server-auth-user') fakeToken = generateFakeToken(authData.user.id, 7 * 24 * 60 * 60) || undefined } // 删除本地 createHmac 逻辑，统一使用 generateFakeToken // 注意：generateFakeToken 已返回完整 64 字符签名

旧版 admin-session cookie 无密码学签名验证
lib/server-admin-auth.ts:26-31  |  requireAdmin 兜底逻辑

admin-session cookie 的 value 直接是 userId 字符串，无 HMAC 签名。攻击者若能猜出任意用户 UUID（短 ID 格式可预测），即可伪造该 cookie 冒充管理员。

// lib/server-admin-auth.ts:26-31 —— 无签名验证的兜底 if (!adminId && adminSession?.value) { if (adminSession.value && adminSession.value.length > 0) { adminId = adminSession.value // ← 直接取 value，无签名验证 } }

解决方案

对 admin-session cookie 也添加 HMAC 签名验证，与新格式一致的验证逻辑：

// lib/server-admin-auth.ts —— 修复后 import { createHmac } from "crypto" const HMAC_SECRET = process.env.HMAC_SECRET function verifyAdminCookieSignature(cookieValue: string): string | null { if (!HMAC_SECRET) return null // 格式：adminId + "_" + expiresAt + "_" + signature(64字符) const parts = cookieValue.split("_") if (parts.length < 3) return null const signature = parts[parts.length - 1] if (!/^[0-9a-f]{64}$/i.test(signature)) return null const payload = parts.slice(0, -1).join("_") const expectedSig = createHmac("sha256", HMAC_SECRET).update(payload).digest("hex") if (signature !== expectedSig) return null const expiresAt = parseInt(parts[parts.length - 2], 10) if (Date.now() / 1000 > expiresAt) return null // 已过期 return parts[0] // 返回 adminId } // 在 requireAdmin 中： if (!adminId && adminSession?.value) { const verifiedId = verifyAdminCookieSignature(adminSession.value) if (verifiedId) adminId = verifiedId }
