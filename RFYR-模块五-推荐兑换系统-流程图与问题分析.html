🎁 月卡兑换规则

RFYR-MONTH-XXXXXX

• 每人免费月卡 1 次（monthly_free_used）

• 付费月卡最多 4 次（monthly_purchase_count）

• 有效期 30 天

• 兑换码生成后 3 天内必须使用

• 不能使用自己生成的兑换码

👑 年卡兑换规则

RFYR-YEAR-XXXXXX

• 不限次数

• 有效期 365 天

• 续期逻辑：setFullYear(+1)，精确1年

• 兑换码生成后 3 天内必须使用

• 不能使用自己生成的兑换码

🔄 续期计算规则

有未过期会员：以现有到期日为基准，顺延月卡+30天/年卡+1年。
已过期会员：从当前日期开始计算。
年卡使用 setFullYear()：精确加1年（如2024-02-29 + 1年 → 2025-02-28）。
月卡使用 setDate()：溢出由JS自动处理（如1月31日+30天 → 3月2日）。

P0-Critical
推荐码格式安全漏洞：缺少服务端长度/正则校验
安全 API
▼
问题描述： /^[a-z0-9]{8}$/ 的格式校验仅在 /api/referral/validate 中实现，但 createReferral() 和 redeemCode() 调用的地方（如注册流程）直接使用用户传入的 code 字符串，没有格式校验。攻击者可构造超长字符串或特殊字符触发数据库错误或绕过业务逻辑。

影响范围： 任何调用 createReferral 的注册流程入口。

根因：格式校验和业务逻辑分离，且只在用户输入校验 API 中做了检查，业务核心函数没有防御性校验。
💡 推荐解决方案：
  • 1
    在 lib/referral.ts 的 createReferral() 入口处添加格式校验：
    // 添加常量 const REFERRAL_CODE_REGEX = /^[a-z0-9]{8}$/ // createReferral 开头添加 if (!REFERRAL_CODE_REGEX.test(referrerCode)) { console.warn('[createReferral] 无效邀请码格式', referrerCode) return }
  • 2
    在 redeemCode() 中对兑换码格式也做预处理，强制 toUpperCase() + 长度校验。
  • 3
    建议：统一邀请码/兑换码格式验证函数，抽取到 lib/utils.ts 供各处复用。
P0-Critical
推荐链深度检查存在 TOCTOU 竞态窗口
安全 数据库
▼
问题描述： 在 createReferral() 中，推荐链深度检查（buildReferralChain）和实际的 INSERT 是两个独立的数据库操作。高并发场景下，两个请求同时通过检查后都执行 INSERT，可能导致推荐链实际深度超过 maxDepth=3 的限制。

影响范围： lib/referral.ts 中的 createReferral() 函数。

根因：读（SELECT）和写（INSERT）之间没有原子性保证，属于经典的 TOCTOU（Time-of-Check-Time-of-Use）漏洞。
💡 推荐解决方案：
  • 1
    在 referrals 表添加 CHECK (depth <= 3) 数据库约束（PostgreSQL 支持）。
    -- 添加深度约束 ALTER TABLE referrals ADD CONSTRAINT referrals_max_depth CHECK (depth <= 3);
  • 2
    在应用层改用单条 UPSERT 语句替代先查后插：
    -- 使用 PostgreSQL UPSERT（ON CONFLICT）配合 depth 约束 WITH depth_calc AS ( SELECT COALESCE(MAX(depth), 0) + 1 AS new_depth FROM referrals WHERE referee_id = $1 ) INSERT INTO referrals (referrer_id, referee_id, depth) SELECT referrer.user_id, refereeId, dc.new_depth FROM referrer_codes rc, depth_calc dc WHERE rc.code = referrerCode AND dc.new_depth <= 3 ON CONFLICT (referee_id) DO NOTHING RETURNING *;
  • 3
    如果 upsert 返回空（0 rows affected），说明深度超限或已有关系。
P1-High
bonus_daily_count 日榜零时差问题：北京时间跨天边界存在竞态
数据库 设计
▼
问题描述： 每日奖励使用 bonus_daily_reset_date 判断是否跨天。当日北京时间为 UTC+8，如果服务器在 UTC 时间 16:00-23:59（北京时间 00:00-07:59）处理请求，toLocalDateString() 返回的日期与 bonus_daily_reset_date 之间的比较可能因时区偏移出现不一致。此外，bonus_daily_reset_date 由 toLocalDateString() 写入，但判断逻辑也用同一函数，存在"恰好跨天"边界条件。

影响范围： lib/referral.ts 中会员奖励逻辑，可能导致某天奖励突然清零或重复累加。
💡 推荐解决方案：
  • 1
    统一使用北京时间（固定 UTC+8），不要依赖服务器本地时区：
    // 统一北京时间计算（固定 UTC+8，不依赖系统时区） function toBeijingDateString(): string { const now = new Date() const beijing = new Date(now.getTime() + 8 * 60 * 60 * 1000) return beijing.toISOString().split('T')[0] // 'YYYY-MM-DD' UTC+8 } // 或使用 Intl.DateTimeFormat const beijingFormatter = new Intl.DateTimeFormat('zh-CN', { timeZone: 'Asia/Shanghai', year: 'numeric', month: '2-digit', day: '2-digit' })
  • 2
    用数据库触发器（PostgreSQL）保证跨天后 bonus_daily_count 自动归零，而不是依赖应用层判断。
  • 3
    在 user_profiles 表加 CHECK (bonus_daily_count >= 0) 约束，防止负数。
P1-High
referrer_article_id 功能未完成：来源文章标记链路断裂
数据库 设计
▼
问题描述： 迁移文件 add-referrer-article-id.sql 为 pending_registrations 表添加了 referrer_article_id 字段，意图是记录被邀请人通过哪个文章页面注册。但：
① captureReferrerFromUrl() 将 referrer_article 存到 localStorage，但没有看到注册流程中读取和使用该值的代码
② pending_registrations 表在用户注册成功后如何传递 referrer_article_id 到 createReferral 的链路不清晰
③ referrer_article_id 存储后没有任何地方消费（查询/使用）

影响范围：数据完整性，无法追踪"哪个文章带来了多少注册转化"，也无法实现"来源文章自动标记已读"功能。
💡 推荐解决方案：
  • 1
    在用户注册 API 中，将 localStorage 中的 referrer_article_id 读取并传入注册逻辑（可通过 X-Header 或请求体）：
    // 在注册表单提交时 const referrerArticle = localStorage.getItem('rfyr_referrer_article') const res = await fetch('/api/register', { body: JSON.stringify({ ..., referrerArticle }) })
  • 2
    注册成功后将该字段写入 pending_registrations（注册完成前）或 referrals 表（注册完成后）：
    -- referrals 表加 referrer_article_id 字段 ALTER TABLE referrals ADD COLUMN IF NOT EXISTS referrer_article_id TEXT; -- 注册成功后写入 UPDATE referrals SET referrer_article_id = $1 WHERE referee_id = $2;
  • 3
    在管理员后台新增"文章转化分析"视图，统计每个 article_id 的推荐注册人数。

P2-Medium
兑换码失效未自动清理：数据库持续膨胀
数据库
▼
问题描述： 兑换码有效期 3 天，过期后状态改为 expired，但没有定时任务或触发器自动将未使用的过期码更新为 expired 状态。已过期的未使用兑换码会持续留在表中，占用存储空间。管理员列表页可以筛选 expired，但不会自动更新。

影响范围：redeem_codes 表数据持续增长，影响查询性能。
💡 推荐解决方案：
  • 1
    添加 PostgreSQL 事件触发器或 cron 任务定期更新过期状态：
    -- 定时任务（pg_cron）或 Supabase Edge Function） UPDATE redeem_codes SET status = 'expired' WHERE status = 'unused' AND expires_at < NOW() AND expires_at > NOW() - INTERVAL '1 day';
  • 2
    在 redeemCode() 查询时添加：.eq('status', 'unused') 避免查到已过期码（当前已做），同时在查询前主动更新一遍最近过期的码。
  • 3
    考虑增加表分区（按月份），简化过期数据清理和查询优化。
P2-Medium

API 设计

问题描述： /api/referral/info 和 /api/referral/stats 都调用同一个 getReferralInfo() 函数，返回几乎相同的数据结构。这是典型的代码重复/接口冗余，增加了维护成本和客户端调用复杂度。

影响范围：前端需要知道调用哪个 API，容易产生混淆；后端维护两个端点但逻辑相同。

  • 2
    删除 app/api/referral/stats/route.ts。
  • 3
    全局搜索调用 /api/referral/stats 的地方，统一改为调用 /api/referral/info。
P3-Low
createReferral 奖励更新失败时未回滚邀请关系
数据库 设计
▼
问题描述： 在 createReferral() 中，邀请关系 INSERT（第104行）和奖励 UPDATE（第153行）是两个独立的数据库操作。如果 INSERT 成功但 UPDATE 失败（网络抖动、超时等），邀请关系已建立，但邀请人拿不到奖励。用户感知上"邀请成功但没奖励"，体验不佳。当前代码只是 console.error 记录日志。

影响范围：邀请奖励可能丢失，影响用户参与邀请活动的积极性。
💡 推荐解决方案：
  • 1
    最佳方案：PostgreSQL 事务（需要 Supabase 支持 Edge Functions 或 RPC）：
    -- 创建 PostgreSQL 存储过程，原子执行： CREATE OR REPLACE FUNCTION create_referral_with_reward(...) RETURNS void AS $$ BEGIN -- 1. 插入邀请关系 INSERT INTO referrals (...) VALUES (...); -- 2. 更新奖励（在同一事务中） UPDATE user_profiles SET bonus_read_count = ... WHERE id = referrer_id; END; $$ LANGUAGE plpgsql;
  • 2
    退而求其次：在 UPDATE 失败时，将奖励补偿写入一张 pending_rewards 补偿表，用后台任务异步补发。
  • 3
    监控告警：当 createReferral 中 updateError 存在时，发送告警通知人工介入。
