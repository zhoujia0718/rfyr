问题分析 · 共 13 项

Critical ×3 · High ×5 · Medium ×3 · Low ×2
CRITICAL

referrer_article_id 字段存储但从未使用

lib/referral.ts

数据库设计了 referrals.referrer_article_id 字段用于追踪用户通过哪篇文章的邀请链接注册，但 createReferral() 函数中从未给该字段赋值，始终为 NULL。这导致无法分析哪些内容渠道带来的转化效果。

解决方案：修改 createReferral() 签名，增加 articleId?: string 参数，并在 INSERT 时写入该字段。UI 侧在调用处传入从 localStorage 读取的 rfyr_referrer_article 值。
P-01 · lib/referral.ts · 字段废弃
CRITICAL

每日重置依赖数据库字段却无定时触发器

lib/referral.ts · supabase migrations

bonus_daily_count 每日重置依赖 bonus_daily_reset_date 字段，但没有任何 Supabase 定时触发器（pg_cron）或后台 Job 主动将该字段归零。当前只能靠 createReferral() 中代码做"被动兜底"：新邀请发生时才检查是否跨天。这导致：

① 如果某天没有任何新邀请，bonus_daily_count 不会重置
② 跨天后第一天第一个邀请人触发检查时，前一天的旧奖励会被加到今天的奖励上
解决方案：添加 Supabase pg_cron 定时任务，每天北京时间 00:05 执行：
UPDATE user_profiles SET bonus_daily_count = 0, bonus_daily_reset_date = CURRENT_DATE
P-02 · 定时任务缺失
CRITICAL

邀请码格式不一致（前端验证 vs 后端生成）

lib/referral-client.ts · app/api/referral/code/route.ts

邀请码验证正则（validate）：/^[a-z0-9]{8}$/（仅小写）
邀请码生成：8位，小写字母+数字混合
但在 validate API 返回"不存在"后，用户看到提示"邀请码不存在"——实际上该邀请码可能只是大写格式。更严重的是：分享链接生成时不做格式校验。

解决方案：在 /api/referral/validate 中，对输入的 code 做 toLowerCase() 归一化后再查询。分享链接生成函数 buildShareUrlWithReferrer 也应同样处理。
P-03 · 格式不兼容
HIGH

内存限流在 Serverless 环境完全失效

app/api/referral/validate/route.ts

当 Supabase rate_limits 查询失败时，降级到内存 Map memoryRateLimitMap。但在 Vercel/Cloudflare Workers 等 Serverless 环境中，每次请求可能落在不同实例，内存状态不共享。攻击者可以通过大量并发请求轻松绕过限流。

解决方案：① 确保 rate_limits 表存在且 RLS 策略正确；② 如果 Supabase 查询失败则直接拒绝请求（返回 503），而不是降级到无效的内存方案；③ 或者改用 Redis 等分布式存储做限流。
P-04 · V-L-05 · Serverless 限流失效
HIGH

referrer_codes 表缺少唯一约束

scripts/create-referral-and-redeem-tables.sql

referrer_codes 表的 code 字段仅有 UNIQUE 约束（无主键），但 id 字段没有主键定义。如果两次并发请求生成了相同 code，第二次 INSERT 可能成功（因为查重逻辑和 INSERT 之间存在 TOCTOU 竞争条件）。

解决方案：添加数据库级唯一约束：
ALTER TABLE referrer_codes ADD CONSTRAINT referrer_codes_code_unique UNIQUE (code);
并将 id 设为 PRIMARY KEY，让碰撞直接触发唯一约束错误（代码中已有最多 10 次重试逻辑兜底）。
P-05 · 数据库约束缺失
HIGH

非会员邀请奖励无上限，存在套利风险

lib/referral.ts

非会员（免费用户）邀请成功后，bonus_read_count 终身累加，无上限。理论上：100 个免费用户各注册 3 个小号，总共获得 600 次额外阅读，等同于无限阅读。这绕过了会员付费的商业模式。

解决方案：对非会员的 bonus_read_count 设置上限（如最多 +10 次），超出后改为发放其他奖励（如体验会员 1 天）；或在 createReferral 中检查邀请人是否仍为非会员状态超过 X 天，若长期为非会员则停止发放奖励。
P-06 · 商业模型漏洞
HIGH

被推荐人奖励完全缺失

lib/referral.ts

架构文档描述"被推荐人：体验会员资格"，但实际代码中完全没有实现。新注册用户（被推荐人）得不到任何奖励（如额外阅读次数或短期会员体验）。这降低了邀请的吸引力，减少了邀请转化率。

解决方案：在 createReferral() 中，被推荐人（新用户）首次注册后，给其增加一次性阅读奖励（如 +3 次）或短期会员体验（如 3 天月度体验）。这需要在 user_profiles 添加新字段 referral_bonus_reads。
P-07 · 奖励缺失
HIGH

Rate limits 清理逻辑异步无保证

app/api/referral/validate/route.ts

V-L-06 修复中使用了 void supabase.from("rate_limits").delete() 做异步清理。这种写法在 Node.js 环境中如果请求提前结束，清理操作可能被中断。且没有错误处理——清理失败不影响主流程，导致 rate_limits 表随时间无限增长。

解决方案：① 使用 Supabase cron job 定时清理（如每天凌晨执行一次完整清理）；② 或者在每次写入时用 DELETE WHERE created_at < now - interval '1 hour'（不依赖异步 void）。
P-08 · V-L-06 · 异步清理无保证
HIGH

管理员可生成兑换码但无记录追踪

app/api/admin/redeem/route.ts

管理员生成兑换码时，created_by 字段存的是"unknown"字符串（当 cookie 解析失败时），无法追溯具体哪个管理员生成了一批兑换码。此外，兑换码没有与特定营销活动绑定的字段，无法做精细化运营分析。

P-10 · 配置不一致
MEDIUM

邀请码验证缺少 CSRF 保护

app/api/referral/validate/route.ts

/api/referral/validate 端点通过 GET 参数或 POST body 接收邀请码，但没有验证请求来源是否来自同源的合法页面。攻击者可以在自己的网站构造表单自动提交大量邀请码进行暴力枚举（虽然有速率限制，但结合 CSRF 可更难追踪）。

解决方案：① 验证 Origin 或 Referer header 是否来自合法域名；② 引入 CSRF token 机制；③ 在速率限制中加入域名白名单逻辑。
P-11 · CSRF 漏洞
MEDIUM

会员升级后邀请奖励类型不转换

lib/referral.ts · lib/reading-limit.ts

假设用户 A 是非会员，邀请人给他 +2 bonus_read_count。之后用户 A 付费成为月度会员。此时用户 A 之前的 bonus_read_count 并不会转换为 bonus_daily_count，导致历史积累的奖励无法在会员期间使用（因为会员使用的是 bonus_daily_count 逻辑）。

解决方案：在会员激活逻辑（/api/membership/activate）中，添加奖励迁移逻辑：
IF bonus_read_count > 0 AND tier == 'monthly'/'yearly': bonus_daily_count += bonus_read_count; bonus_read_count = 0
