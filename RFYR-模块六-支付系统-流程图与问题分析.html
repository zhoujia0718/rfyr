🔴 严重问题（影响资金安全和数据完整性）

严重 P-01: 支付完全依赖模拟，无真实支付网关

位置：components/payment-dialog.tsx:44-48 和 :98-134

描述：二维码图片是 /placeholder.svg 占位图，"模拟已扫码支付"按钮直接 setTimeout(3000) 然后调用激活 API。没有任何真实的微信支付/支付宝集成。

// payment-dialog.tsx:44-48
const mockQRCode = {
  wechat: "/placeholder.svg",  // ← 占位图
  alipay: "/placeholder.svg",
}
// payment-dialog.tsx:98-134
const simulatePayment = () => {
  setPaymentStatus('scanning')
  setTimeout(async () => {          // ← 3秒后直接成功
    setPaymentStatus('success')
    await fetch("/api/membership/activate", { ... })
  }, 3000)
}

影响：用户无法真正付费，平台无法获得真实收入。整个支付系统无法生产使用。

解决建议：

  • 接入微信支付 V3 API（https://api.mch.weixin.qq.com/v3/pay/transactions/native）
  • 使用官方 SDK 生成真正的支付二维码
  • 配置 WECHAT_APP_ID、WECHAT_MCH_ID、WECHAT_API_KEY 等环境变量
  • 移除 simulatePayment 函数，改为调用真实的统一下单 API
严重 P-02: activate API 无事务保护，可能导致数据不一致

位置：app/api/membership/activate/route.ts:118-157

描述：代码分两步执行：①写入 memberships 表 ②更新 users.vip_tier。这两步不在同一个数据库事务中，如果第二步失败（如 RLS 策略阻止），第一步已提交无法回滚。

// 步骤1：写入 memberships（成功）
await supabase.from("memberships").insert({ user_id, membership_type: vipTier, ... })

// 步骤2：更新 users.vip_tier（可能失败！）
const { error: userError } = await supabase
  .from("users")
  .update({ vip_tier: vipTier })
  .eq("id", userId)
// ⚠ 如果 userError 不为空，但 memberships 已经写入了！
// 会员表有记录，但 users.vip_tier 没更新 → 前端读 vip_tier 判断为 "none"

影响：用户支付成功但会员未激活，引发客诉。数据库状态处于中间不一致状态。

解决建议：

  • 使用 Supabase RPC 函数包装整个激活逻辑为原子事务
  • 或使用 PostgreSQL 事务（BEGIN/COMMIT/ROLLBACK）
  • 激活前先检查所有依赖是否满足，原子执行
严重 P-03: 订单号前端生成，可被枚举猜测

位置：components/payment-dialog.tsx:59-66

描述：订单号由前端通过 Date.now() + Math.random() 生成。这种伪随机数在客户端可被预测，攻击者可枚举订单号调用激活 API。

// payment-dialog.tsx:59-66
const newOrderId = `ORD${Date.now()}${Math.random().toString(36).substr(2, 4).toUpperCase()}`
// 可被枚举：ORD1745xxxxXXYZ → 尝试激活 API

影响：activate API 虽然检查订单归属，但 orders 表记录由前端创建的攻击面仍然存在。

解决建议：

  • 订单号必须由服务端生成（调用支付网关后获得）
  • 或使用 crypto.randomUUID() + 服务端存储
  • activate API 应强制要求订单在 payments 表中存在且状态为 paid

🟡 中等问题（功能缺陷和数据不一致风险）

中等 P-04: vip_tier 枚举值不统一（存储 vs 读取）

位置：app/api/membership/activate/route.ts:127 和 components/membership-provider.tsx:68-70, 113-115, 173-175

描述：activate API 写入 yearly_vip 和 monthly_vip，但 status API 返回同样的值后，前端使用 includes("yearly") 来判断类型。

// 后端写入 (activate/route.ts:127)
const vipTier = planType === "yearly" ? "yearly_vip" : "monthly_vip"
await supabase.from("memberships").insert({ membership_type: vipTier, ... })
await supabase.from("users").update({ vip_tier: vipTier, ... })

// 前端读取 (membership-provider.tsx:68-70)
const vipTier = data.vip_tier  // "yearly_vip"
const type = vipTier.includes("yearly") ? "yearly" : "monthly"
// ⚠ 脆弱的字符串匹配。如果未来出现 "yearly_pro" 会误判

解决建议：

  • 统一使用枚举值：前端用 "yearly"，后端也写 "yearly"
  • 或创建 MembershipTier TypeScript 类型约束
  • 避免使用 includes() 做类型判断，改用精确匹配
中等 P-05: 激活 API 无幂等性保护，重复调用会重复激活

位置：app/api/membership/activate/route.ts

描述：activate API 虽然检查 paymentRow.status === "approved"（payments 表存在时），但对于兑换码路径（payments 表无记录）完全没有保护。如果用户在激活成功提示显示之前刷新页面，refreshMembership() 再次触发激活逻辑。

// activate/route.ts:79-91 - 只有 payments 表记录存在时检查
if (paymentRow) {
  if (paymentRow.status === "approved") {
    return NextResponse.json({ error: "订单已激活" }, { status: 409 })
  }
}
// ⚠ 当 payments 表无记录时（兑换码路径），完全没有检查！
// 重复调用会多次写入 memberships 表

解决建议：

  • 在 memberships 表上增加 unique constraint：(user_id, membership_type, order_id)
  • 或添加已激活订单 ID 的 Redis/In-memory 缓存（带 TTL）
  • activate 前先查询是否已存在活跃会员，有则返回已有记录
中等 P-06: 免费月卡完全依赖人工操作，无自动化

位置：components/wechat-guide-overlay.tsx:126-138

描述：用户扫码完成微信转发任务后，管理员需要手动在后台为用户激活月卡。没有任何自动化验证机制（如网页截图识别、公众号消息回调）。

// wechat-guide-overlay.tsx - 显示二维码后无后续
<img src="/qrcode/微信图片_20260328173325_3_11.png" />
// 用户完成转发后 → 管理员手动激活（无系统通知）

解决建议：

  • 接入微信客服消息 API，当用户发送特定关键词时自动激活
  • 或使用微信开放平台网页授权 + 后端记录，验证用户身份后自动激活
  • 增加任务状态通知（站内消息/邮件）
中等 P-07: 续费逻辑存在边界条件缺陷

位置：app/api/membership/activate/route.ts:101-116

描述：续费逻辑在计算 finalEndDate 时只检查 existingMembership.end_date > now()，但如果用户有一个已过期的会员记录（status=expired），续费会创建新记录而非追加到当前。

// activate/route.ts:102-116
const { data: existingMembership } = await supabase
  .from("memberships")
  .select("end_date, membership_type")
  .eq("user_id", userId)
  .eq("status", "active")  // ← 只查 active，过期的完全忽略
  .maybeSingle()

if (existingMembership) {
  // 从当前到期日开始追加
  finalEndDate = new Date(existingEnd.getTime() + days * 24 * 60 * 60 * 1000)
}

解决建议：

  • 改为查询最近一条 membership 记录（不限 status）
  • 根据最后到期日计算续费起始点
  • 或设计更合理的会员有效期管理策略
中等 P-08: activateMembership 的 catch 分支绕过 API 直接写 localStorage

位置：components/membership-provider.tsx:189-222

描述：activateMembership 函数在 API 调用失败时会 fallback 到 createMembership() 并直接写入 localStorage，完全绕过服务端验证。这在网络异常时会导致用户看到虚假的会员状态。

// membership-provider.tsx:189-222
try {
  const res = await fetch("/api/membership/activate", { ... })
  if (res.ok) { await refreshMembership() }
} catch {
  // ⚠ 网络异常时直接写本地，不经过服务端！
  const m = createMembership(type, days)  // 纯前端计算
  saveMembershipToStorage(m)             // 直接存 localStorage
  setMembership(m)
}

解决建议：

  • catch 分支改为重试机制（如指数退避重试 3 次）
  • 如果最终失败，显示错误提示而非虚假成功
  • 考虑使用 SWR/React Query 的乐观更新模式

🟢 轻微问题（代码质量和体验优化）

轻微 P-09: RedeemDialog 关闭时清空状态，但无防抖保护

位置：components/redeem-dialog.tsx:109-116

描述：用户快速输入兑换码时，如果网络较慢，用户可能在请求进行中关闭弹窗，然后重新打开并再次提交，导致重复请求。

// redeem-dialog.tsx:109-116
const handleOpenChange = (val: boolean) => {
  if (!val) {
    setCode("")         // 关闭时清空
    setStatus("idle")
    setMessage("")
  }
  onOpenChange(val)
  // ⚠ 没有 abort controller 取消进行中的请求

解决建议：使用 AbortController 取消进行中的请求，或在提交时禁用弹窗关闭。

轻微 P-10: 支付对话框 5 分钟倒计时无服务端验证

位置：components/payment-dialog.tsx:54

描述：倒计时完全在前端运行，服务端不知道订单的有效期。攻击者可绕过前端倒计时，即使订单已超时仍调用激活 API（虽然后端目前没有强验证）。

// payment-dialog.tsx:54, 68-88
const [countdown, setCountdown] = useState(300) // 5分钟倒计时
// 仅前端计时，后端不知道订单何时创建

解决建议：在 payments 表或 orders 表增加 expires_at 字段，activate API 验证当前时间是否在有效期内。

轻微 P-11: membership-provider 初始化时两次 API 调用

位置：components/membership-provider.tsx:35-85

描述：初始化逻辑同时执行：①读取 localStorage ②调用 GET /api/membership/status。这导致每次页面加载都会产生一个不必要的网络请求，即使缓存有效。

// membership-provider.tsx:35-85
const init = async () => {
  const stored = getMembershipFromStorage()   // 第1步：读缓存
  if (stored && isMembershipValid(stored)) {
    setMembership(stored)                     // ← 立即显示缓存
  }
  // 第2步：无论缓存是否有效，都发 API 请求
  const res = await fetch("/api/membership/status", { ... })
  // → 浪费网络请求，但有"静默刷新"的合理性

解决建议：考虑使用 SWR 或 React Query，它们天然支持"缓存优先，后台刷新"模式，减少不必要的请求同时保证数据新鲜。
